# 📘 Go | Day 13 — указатели и value semantics: копии, адреса, `nil`, безопасное изменение данных

---

## 📖 Теория

### 1. Зачем после `error` и `defer` разбирать указатели

На прошлых уроках ты уже работал с:

* переменными
* `array`
* `slice`
* `map`
* `string`
* функциями
* несколькими возвращаемыми значениями
* пакетами
* ошибками
* `defer`

Теперь пора явно разобрать вопрос, который уже несколько раз был рядом:

* когда значение копируется
* когда изменения видны снаружи
* когда нужно вернуть новое значение
* когда можно передать адрес существующего значения

В Go это называется `value semantics`.

`value semantics` означает:

* значения обычно передаются и присваиваются как копии
* переменная содержит значение
* при присваивании или передаче в функцию создаётся новая переменная с копией старого значения

Но рядом есть важный инструмент — указатель.

Указатель хранит не само значение, а адрес значения.

Главная идея урока:

* понять, где Go делает копию
* научиться читать и писать типы вроде `*int`
* безопасно менять значение через указатель
* не путать указатель с `slice` и `map`
* подготовиться к `struct` и методам, где указатели будут встречаться постоянно

Короткий словарь:

* `value` — значение, например `10`, `"go"` или `[]int{1, 2, 3}`
* `variable` — переменная, то есть имя, под которым хранится значение
* `copy` — копия значения
* `address` — адрес места, где лежит значение
* pointer / указатель — значение, которое хранит адрес другого значения
* `&x` — взять адрес переменной `x`
* `*int` — тип “указатель на `int`”
* `*p` — разыменовать указатель `p`, то есть получить значение по адресу
* `nil pointer` — указатель, который ни на что не указывает
* `dereference` / разыменование — доступ к значению через указатель

### 2. Обычное присваивание копирует значение

Начнём без указателей.

```go
package main

import "fmt"

func main() {
	a := 10
	b := a

	b = 20

	fmt.Println(a)
	fmt.Println(b)
}
```

Результат:

```text
10
20
```

Почему:

* `a := 10` создаёт переменную `a`
* `b := a` создаёт новую переменную `b`
* в `b` попадает копия значения из `a`
* изменение `b` не меняет `a`

Что здесь важно:

* `b` — не второе имя для `a`
* `b` — отдельная переменная
* у неё своё значение

---

### 3. Параметры функции тоже получают копии

Передача аргумента в функцию похожа на присваивание.

```go
package main

import "fmt"

func change(n int) {
	n = 100
}

func main() {
	value := 10

	change(value)

	fmt.Println(value)
}
```

Результат:

```text
10
```

Почему:

* `change(value)` передаёт в функцию копию `value`
* параметр `n` — отдельная локальная переменная внутри функции
* `n = 100` меняет только эту локальную копию
* исходная переменная `value` в `main` остаётся равной `10`

Это базовое правило Go:

* аргументы функций передаются по значению
* то есть в параметр попадает копия

---

### 4. Если нужно изменить значение снаружи, можно вернуть новое значение

Часто самый простой и понятный способ — не использовать указатель, а вернуть новое значение.

```go
package main

import "fmt"

func changed(n int) int {
	return 100
}

func main() {
	value := 10

	value = changed(value)

	fmt.Println(value)
}
```

Результат:

```text
100
```

Почему:

* функция возвращает новое значение
* `value = changed(value)` явно записывает результат обратно в `value`
* изменение видно снаружи, потому что мы сами присвоили новый результат

Что здесь важно:

* это часто самый читаемый вариант
* особенно для маленьких значений вроде `int`, `string`, `bool`
* указатель нужен не всегда

---

### 5. Оператор `&`: взять адрес переменной

Теперь посмотрим на адрес.

```go
package main

import "fmt"

func main() {
	value := 10
	p := &value

	fmt.Println(value)
	fmt.Println(p)
}
```

Один из возможных результатов:

```text
10
0xc0000120c0
```

Почему:

* `value` хранит число `10`
* `&value` берёт адрес переменной `value`
* `p` хранит этот адрес
* конкретный адрес может отличаться при каждом запуске

Что здесь важно:

* адрес — это техническое место значения в памяти
* точное число адреса обычно не важно
* важно, что через этот адрес можно обратиться к исходной переменной

---

### 6. Тип `*int`: указатель на `int`

Если `value` имеет тип `int`, то `&value` имеет тип `*int`.

```go
package main

import "fmt"

func main() {
	value := 10
	p := &value

	fmt.Printf("%T\n", value)
	fmt.Printf("%T\n", p)
}
```

Результат:

```text
int
*int
```

Почему:

* `value` — обычное число типа `int`
* `p` — указатель на число типа `int`
* запись `*int` читается как “pointer to int”, то есть “указатель на int”

Похожая логика работает и для других типов:

```go
var name string
var namePointer *string

var ok bool
var okPointer *bool
```

Этот фрагмент не нужно запускать отдельно.

Он показывает форму типов:

* `*string` — указатель на `string`
* `*bool` — указатель на `bool`
* `*[]int` — указатель на `slice` типа `[]int`, но такой тип нужен гораздо реже, чем может показаться

---

### 7. Оператор `*`: получить значение по адресу

Если `p` хранит адрес, то `*p` даёт значение по этому адресу.

```go
package main

import "fmt"

func main() {
	value := 10
	p := &value

	fmt.Println(*p)
}
```

Результат:

```text
10
```

Почему:

* `p` указывает на `value`
* `*p` означает “возьми значение там, куда указывает `p`”
* там лежит `10`

Здесь символ `*` используется в двух похожих, но разных местах:

* `*int` в типе означает “указатель на `int`”
* `*p` в выражении означает “значение по указателю `p`”

---

### 8. Через указатель можно изменить исходное значение

Указатель позволяет менять значение, на которое он указывает.

```go
package main

import "fmt"

func main() {
	value := 10
	p := &value

	*p = 20

	fmt.Println(value)
	fmt.Println(*p)
}
```

Результат:

```text
20
20
```

Почему:

* `p` хранит адрес переменной `value`
* `*p = 20` записывает `20` по этому адресу
* поэтому меняется сама переменная `value`

Что здесь важно:

* `p = ...` меняет сам указатель
* `*p = ...` меняет значение, на которое указатель смотрит

---

### 9. Функция может изменить значение через указатель

Теперь вернёмся к функции.

```go
package main

import "fmt"

func change(n *int) {
	*n = 100
}

func main() {
	value := 10

	change(&value)

	fmt.Println(value)
}
```

Результат:

```text
100
```

Почему:

* `change` принимает `*int`, то есть указатель на `int`
* `change(&value)` передаёт адрес переменной `value`
* внутри функции `n` тоже является копией, но копией адреса
* `*n = 100` меняет значение по этому адресу
* поэтому исходная переменная `value` в `main` меняется

Это важная мысль:

* в Go всё равно передаётся копия
* но если копируется адрес, то обе копии адреса ведут к одному значению

---

### 10. Указатель тоже копируется

Указатель — это тоже значение.

Его можно скопировать в другую переменную.

```go
package main

import "fmt"

func main() {
	value := 10

	p1 := &value
	p2 := p1

	*p2 = 30

	fmt.Println(value)
	fmt.Println(*p1)
	fmt.Println(*p2)
}
```

Результат:

```text
30
30
30
```

Почему:

* `p1` хранит адрес `value`
* `p2 := p1` копирует этот адрес
* теперь `p1` и `p2` указывают на одну и ту же переменную
* запись через `*p2` видна при чтении через `*p1`

Что здесь важно:

* копия указателя — это не копия исходного числа
* это копия адреса
* две переменные-указателя могут смотреть на одно и то же значение

---

### 11. `nil pointer`: указатель, который никуда не указывает

У указателя есть zero value.

Для указателя это `nil`.

```go
package main

import "fmt"

func main() {
	var p *int

	fmt.Println(p == nil)
}
```

Результат:

```text
true
```

Почему:

* `var p *int` создаёт указатель без адреса
* zero value для указателя — `nil`
* `p == nil` показывает, что указатель никуда не указывает

Что здесь важно:

* `nil` для указателя означает “адреса нет”
* перед разыменованием указателя часто нужно понимать, может ли он быть `nil`

---

### 12. Нельзя разыменовывать `nil`

Если указатель равен `nil`, у него нет значения, к которому можно обратиться.

```go
package main

func main() {
	var p *int

	_ = *p
}
```

Один из возможных результатов:

```text
panic: runtime error: invalid memory address or nil pointer dereference
```

Почему:

* `p` равен `nil`
* `*p` пытается получить значение по адресу
* адреса нет
* программа падает с `panic`

`panic` уже был в прошлом уроке.

Здесь он появляется как результат ошибки программиста:

* программа попыталась обратиться туда, куда указатель не указывает

Безопасный вариант:

```go
package main

import "fmt"

func printValue(p *int) {
	if p == nil {
		fmt.Println("no value")
		return
	}

	fmt.Println(*p)
}

func main() {
	var p *int
	value := 10

	printValue(p)
	printValue(&value)
}
```

Результат:

```text
no value
10
```

Почему:

* первый вызов получает `nil`
* функция делает ранний выход через `return`
* второй вызов получает настоящий адрес
* поэтому `*p` безопасен

---

### 13. Указатели можно сравнивать

Указатели можно сравнивать с `nil`.

Ещё можно проверить, указывают ли два указателя на одно и то же место.

```go
package main

import "fmt"

func main() {
	a := 10
	b := 10

	p1 := &a
	p2 := &a
	p3 := &b

	fmt.Println(p1 == p2)
	fmt.Println(p1 == p3)
}
```

Результат:

```text
true
false
```

Почему:

* `p1` и `p2` указывают на одну переменную `a`
* `p3` указывает на другую переменную `b`
* значения `a` и `b` одинаковые, но адреса разные

Что здесь важно:

* сравнение указателей сравнивает адреса
* оно не сравнивает значения внутри

---

### 14. В Go нет арифметики указателей

В некоторых языках через указатель можно “двигаться” по памяти.

В обычном Go так нельзя.

Этот фрагмент не должен компилироваться:

```go
package main

func main() {
	value := 10
	p := &value

	p = p + 1
}
```

Один из возможных текстов ошибки компиляции:

```text
invalid operation: p + 1 (mismatched types *int and untyped int)
```

Почему:

* `p` имеет тип `*int`
* `1` — число
* Go не разрешает складывать указатель и число

Что здесь важно:

* указатели в Go нужны для явного доступа к существующему значению
* они не используются как инструмент ручного обхода памяти
* для обхода коллекций используй `for`, индексы и `range`

### 15. `array` копируется целиком

Это уже было видно в уроках про `array`, но теперь можно назвать причину точнее.

`array` — значение.

При присваивании он копируется целиком.

```go
package main

import "fmt"

func main() {
	a := [3]int{1, 2, 3}
	b := a

	b[0] = 100

	fmt.Println(a)
	fmt.Println(b)
}
```

Результат:

```text
[1 2 3]
[100 2 3]
```

Почему:

* `b := a` создаёт копию массива
* `b[0] = 100` меняет только копию
* исходный массив `a` не меняется

Функция с массивом ведёт себя так же:

```go
package main

import "fmt"

func change(a [3]int) {
	a[0] = 100
}

func main() {
	nums := [3]int{1, 2, 3}

	change(nums)

	fmt.Println(nums)
}
```

Результат:

```text
[1 2 3]
```

Почему:

* параметр `a` внутри `change` — копия массива
* изменение элемента происходит в копии
* массив из `main` остаётся прежним

---

### 16. Указатель на `array` позволяет изменить исходный массив

Можно передать указатель на массив.

```go
package main

import "fmt"

func change(a *[3]int) {
	(*a)[0] = 100
}

func main() {
	nums := [3]int{1, 2, 3}

	change(&nums)

	fmt.Println(nums)
}
```

Результат:

```text
[100 2 3]
```

Почему:

* `&nums` берёт адрес массива
* функция получает указатель на массив
* `(*a)[0] = 100` меняет первый элемент исходного массива

В Go для массива можно писать короче:

```go
a[0] = 100
```

вместо:

```go
(*a)[0] = 100
```

Но в этом уроке полезно увидеть полную форму.

Что здесь важно:

* указатель на массив встречается реже, чем `slice`
* чаще для коллекций используют `slice`
* но пример хорошо показывает идею: через указатель меняется исходное значение

### 17. `slice` копируется, но его элементы могут быть общими

`slice` — тоже значение.

Но значение `slice` — это не все элементы целиком.

Практически его удобно понимать как маленькое описание:

* где начинаются данные
* какая длина
* какая вместимость

При присваивании копируется это описание.

Сами элементы могут остаться общими.

```go
package main

import "fmt"

func main() {
	a := []int{1, 2, 3}
	b := a

	b[0] = 100

	fmt.Println(a)
	fmt.Println(b)
}
```

Результат:

```text
[100 2 3]
[100 2 3]
```

Почему:

* `b := a` копирует значение `slice`
* обе копии `slice` смотрят на один и тот же underlying array
* изменение элемента через `b[0]` видно через `a[0]`

Это не отменяет value semantics.

Просто копия `slice` содержит ссылку на те же элементы.

---

### 18. Функция может изменить элементы `slice` без указателя

Если функция получает `slice`, она получает копию slice-описания.

Но элементы обычно остаются общими.

```go
package main

import "fmt"

func change(nums []int) {
	nums[0] = 100
}

func main() {
	values := []int{1, 2, 3}

	change(values)

	fmt.Println(values)
}
```

Результат:

```text
[100 2 3]
```

Почему:

* параметр `nums` — копия `slice`
* эта копия смотрит на те же элементы
* `nums[0] = 100` меняет общий элемент

Что здесь важно:

* для изменения элементов `slice` обычно не нужен `*[]int`
* `[]int` уже достаточно

### 19. Но изменение длины `slice` нужно вернуть наружу

Теперь важный edge case.

`append` может вернуть новый `slice`.

Если сделать `append` внутри функции и не вернуть результат, внешний `slice` может не получить новую длину.

```go
package main

import "fmt"

func add(nums []int) {
	nums = append(nums, 4)
	fmt.Println("inside:", nums)
}

func main() {
	values := []int{1, 2, 3}

	add(values)

	fmt.Println("outside:", values)
}
```

Результат:

```text
inside: [1 2 3 4]
outside: [1 2 3]
```

Почему:

* параметр `nums` — копия `slice`
* `append` возвращает новое значение `slice`
* это новое значение записывается только в локальную переменную `nums`
* переменная `values` в `main` не меняет свою длину

Правильный вариант:

```go
package main

import "fmt"

func add(nums []int) []int {
	return append(nums, 4)
}

func main() {
	values := []int{1, 2, 3}

	values = add(values)

	fmt.Println(values)
}
```

Результат:

```text
[1 2 3 4]
```

Почему:

* функция возвращает новый `slice`
* `values = add(values)` сохраняет результат
* внешняя переменная теперь видит новую длину

Это один из самых важных практических выводов урока.

### 20. `map` копируется как значение, но данные остаются общими

`map` тоже передаётся по значению.

Но копия `map` указывает на те же внутренние данные.

```go
package main

import "fmt"

func change(scores map[string]int) {
	scores["go"] = 100
}

func main() {
	values := map[string]int{
		"go": 10,
	}

	change(values)

	fmt.Println(values["go"])
}
```

Результат:

```text
100
```

Почему:

* параметр `scores` — копия значения `map`
* копия всё равно работает с теми же внутренними данными
* запись по ключу видна снаружи

Что здесь важно:

* для добавления, удаления и изменения элементов `map` обычно не нужен `*map`
* обычного `map[string]int` достаточно

### 21. Если нужно заменить весь `map`, верни новый `map`

Изменить элементы существующего `map` можно без указателя.

Но если функция создаёт совсем новый `map`, лучше вернуть его.

```go
package main

import "fmt"

func reset(scores map[string]int) {
	scores = map[string]int{
		"go": 100,
	}
}

func main() {
	values := map[string]int{
		"go": 10,
	}

	reset(values)

	fmt.Println(values["go"])
}
```

Результат:

```text
10
```

Почему:

* параметр `scores` — локальная копия значения `map`
* `scores = map[string]int{...}` переназначает только локальную переменную
* переменная `values` в `main` продолжает указывать на старый `map`

Правильный вариант:

```go
package main

import "fmt"

func reset(scores map[string]int) map[string]int {
	return map[string]int{
		"go": 100,
	}
}

func main() {
	values := map[string]int{
		"go": 10,
	}

	values = reset(values)

	fmt.Println(values["go"])
}
```

Результат:

```text
100
```

Почему:

* функция возвращает новый `map`
* `values = reset(values)` явно сохраняет результат
* теперь `values` указывает на новый `map`

---

### 22. `string` тоже значение, но его нельзя менять по символам

`string` в Go — значение.

При присваивании переменная получает своё значение строки.

Но строку нельзя менять по индексу.

```go
package main

func main() {
	text := "go"

	text[0] = 'G'
}
```

Один из возможных текстов ошибки компиляции:

```text
cannot assign to text[0]
```

Почему:

* строки в Go immutable, то есть неизменяемые
* `text[0]` можно прочитать
* но нельзя записать новое значение в байт строки

Правильный подход уже был в уроке про `string`, `byte` и `rune`:

* создать новое значение
* собрать новую строку через `[]rune`, `[]byte` или функции из `strings`

### 23. Когда указатель нужен, а когда лучше вернуть значение

Указатель уместен, когда:

* функция должна изменить уже существующую переменную
* нужно явно показать, что функция меняет переданное значение
* значение большое и копировать его нежелательно
* позже это будет метод с pointer receiver

Вернуть новое значение часто лучше, когда:

* значение маленькое
* функция похожа на преобразование
* старое значение не нужно менять на месте
* так код проще читать

Пример с возвратом нового значения:

```go
package main

import "fmt"

func double(n int) int {
	return n * 2
}

func main() {
	value := 10
	value = double(value)

	fmt.Println(value)
}
```

Результат:

```text
20
```

Пример с изменением через указатель:

```go
package main

import "fmt"

func doubleInPlace(n *int) {
	*n = *n * 2
}

func main() {
	value := 10
	doubleInPlace(&value)

	fmt.Println(value)
}
```

Результат:

```text
20
```

Оба варианта рабочие.

Но смысл разный:

* `double` возвращает новое значение
* `doubleInPlace` меняет существующую переменную

### 24. Указатель может быть сигналом “значение необязательное”

Иногда указатель используют, чтобы отличить:

* значение есть
* значения нет

```go
package main

import "fmt"

func printLimit(limit *int) {
	if limit == nil {
		fmt.Println("no limit")
		return
	}

	fmt.Println("limit:", *limit)
}

func main() {
	max := 10

	printLimit(nil)
	printLimit(&max)
}
```

Результат:

```text
no limit
limit: 10
```

Почему:

* `nil` означает, что значения нет
* `&max` означает, что значение есть и оно равно `10`

Что здесь важно:

* это не единственный способ описывать необязательность
* иногда лучше подойдёт `(value, ok)` или `(value, error)`
* выбор зависит от смысла задачи

Сравнение:

* `(value, ok)` хорошо подходит, когда отсутствие — нормальный ожидаемый случай
* `(value, error)` подходит, когда что-то пошло не так
* `*int` может подойти, когда нужно различить “нет значения” и “есть значение `0`”

### 25. Адрес элемента `slice`

Иногда нужно получить указатель на конкретный элемент `slice`.

Для этого удобно использовать индекс.

```go
package main

import "fmt"

func main() {
	nums := []int{10, 20, 30}

	p := &nums[1]
	*p = 200

	fmt.Println(nums)
}
```

Результат:

```text
[10 200 30]
```

Почему:

* `&nums[1]` берёт адрес второго элемента
* `*p = 200` меняет этот элемент
* `slice` печатается уже с новым значением

Важная привычка:

* если нужен адрес элемента `slice`, бери его по индексу: `&items[i]`
* так ясно видно, что указатель относится именно к элементу коллекции

### 26. Мостик к `struct` и методам

Следующий урок будет про `struct`.

Там указатели станут ещё важнее.

Почему:

* `struct` — это значение
* при передаче в функцию `struct` тоже копируется
* если нужно изменить поля исходной структуры, часто используют указатель на структуру
* в уроке про методы появятся `value receiver` и `pointer receiver`

Пока не нужно глубоко уходить в `struct`.

Достаточно вынести из этого урока базовую модель:

* обычное значение копируется
* указатель хранит адрес
* через указатель можно изменить исходное значение
* `nil`-указатель нельзя разыменовывать
* `slice` и `map` тоже передаются по значению, но их внутренние данные часто остаются общими

---

## 🔎 Что особенно важно понять

* В Go аргументы функций передаются по значению.
* Передача по значению означает, что параметр функции получает копию.
* Указатель — это тоже значение, и оно тоже копируется.
* Копия указателя ведёт к тому же исходному значению.
* `&x` берёт адрес переменной `x`.
* `*p` получает или меняет значение по адресу, который хранится в `p`.
* `nil`-указатель никуда не указывает.
* Разыменование `nil` приводит к `panic`.
* `array` копируется целиком.
* `slice` копируется как маленькое описание, но элементы обычно остаются общими.
* `map` копируется как значение, но записи внутри обычно остаются общими.
* `append` внутри функции нужно возвращать наружу, если снаружи должна измениться длина `slice`.

## ⚠️ Типичные ошибки

### Ошибка 1. Думать, что функция меняет `int` без указателя или возврата

Неверная интуиция:

```go
func change(n int) {
	n = 100
}
```

Такая функция меняет только локальную копию.

Если нужно изменить внешнюю переменную, выбери один из вариантов:

* вернуть новое значение
* передать указатель

---

### Ошибка 2. Путать `p = ...` и `*p = ...`

`p = ...` меняет сам указатель.

`*p = ...` меняет значение по указателю.

```go
value := 10
p := &value

*p = 20
```

После этого `value` станет `20`.

---

### Ошибка 3. Разыменовывать указатель без понимания, может ли он быть `nil`

Опасно:

```go
func printValue(p *int) {
	fmt.Println(*p)
}
```

Если `p == nil`, программа упадёт с `panic`.

Безопаснее:

```go
func printValue(p *int) {
	if p == nil {
		fmt.Println("no value")
		return
	}

	fmt.Println(*p)
}
```

---

### Ошибка 4. Думать, что `slice` полностью копируется при присваивании

```go
a := []int{1, 2, 3}
b := a
b[0] = 100
```

После этого `a[0]` тоже будет `100`.

Почему:

* копируется значение `slice`
* элементы остаются общими

Если нужна независимая копия элементов, используй `copy` или `slices.Clone`, которые уже были в прошлых уроках.

---

### Ошибка 5. Делать `append` внутри функции и ждать, что длина внешнего `slice` изменится сама

Опасная версия:

```go
func add(nums []int) {
	nums = append(nums, 10)
}
```

Лучше:

```go
func add(nums []int) []int {
	return append(nums, 10)
}
```

И вызывать так:

```go
nums = add(nums)
```

---

### Ошибка 6. Передавать `*[]int` там, где достаточно `[]int`

Для изменения элементов `slice` обычно достаточно обычного `[]int`.

```go
func normalize(nums []int) {
	for i := range nums {
		if nums[i] < 0 {
			nums[i] = 0
		}
	}
}
```

Указатель на `slice` нужен редко.

На текущем этапе почти всегда лучше:

* менять элементы через `[]int`
* возвращать новый `slice`, если меняется длина

---

### Ошибка 7. Использовать указатель, когда обычный возврат значения проще

Не нужно превращать каждый `int` в `*int`.

Если функция просто считает новое значение, обычный `return` часто лучше.

```go
func next(n int) int {
	return n + 1
}
```

Такой код легче читать, тестировать и использовать.

---

### Ошибка 8. Сравнивать указатели и думать, что сравниваются значения внутри

```go
a := 10
b := 10

fmt.Println(&a == &b)
```

Результат:

```text
false
```

Почему:

* значения одинаковые
* переменные разные
* адреса разные

---

## 🧠 Полезные паттерны

### 1. Вернуть новое значение для чистого преобразования

Если функция не обязана менять исходную переменную, верни результат.

```go
func trimAndLower(text string) string {
	text = strings.TrimSpace(text)
	return strings.ToLower(text)
}
```

Этот фрагмент использует `strings`, который уже был в прошлых уроках.

Паттерн:

* получил значение
* посчитал новое
* вернул результат

---

### 2. Передать указатель, если функция должна изменить конкретную переменную

```go
func reset(n *int) {
	*n = 0
}
```

Паттерн:

* функция принимает `*int`
* вызывающий код явно передаёт `&value`
* по вызову видно, что функция может изменить переменную

---

### 3. Проверять `nil` через guard clause

```go
func printValue(p *int) {
	if p == nil {
		fmt.Println("no value")
		return
	}

	fmt.Println(*p)
}
```

Паттерн:

* сначала проверяется плохой или пустой случай
* затем идёт основной путь
* это продолжение привычки `early return` из прошлого урока

---

### 4. Для `append` возвращать новый `slice`

```go
func addPositive(nums []int, value int) []int {
	if value <= 0 {
		return nums
	}

	return append(nums, value)
}
```

Паттерн:

* `slice` приходит как аргумент
* функция возвращает итоговый `slice`
* вызывающий код сохраняет результат

---

### 5. Для изменения элементов `slice` принимать обычный `slice`

```go
func replaceNegativeWithZero(nums []int) {
	for i := range nums {
		if nums[i] < 0 {
			nums[i] = 0
		}
	}
}
```

Паттерн:

* функция не меняет длину
* функция меняет элементы
* указатель на `slice` не нужен

---

### 6. Для адреса элемента коллекции использовать индекс

```go
p := &nums[i]
```

Паттерн:

* если нужен указатель на элемент `slice`, бери адрес через индекс
* так код явно связан с конкретной позицией в коллекции

---

## 🧠 Сравнение с практикой backend

В backend-коде указатели встречаются постоянно, но не как “сложная магия”.

Чаще всего они нужны для простых вещей:

* изменить уже существующее значение
* передать большую модель без лишнего копирования
* отличить “поле отсутствует” от “поле есть, но равно zero value”
* работать с методами, которые меняют состояние объекта
* аккуратно передавать данные между слоями программы

Но хорошая привычка такая:

* не использовать указатель автоматически
* сначала понять смысл владения и изменения
* если функция просто преобразует данные, вернуть новое значение
* если функция должна менять существующее значение, указатель может быть уместен

Слово “владение” здесь можно понимать просто:

* кто отвечает за текущее значение
* кто имеет право его менять
* должен ли вызывающий код увидеть изменение

Глубокие правила памяти, escape analysis и внутренности runtime пока не нужны.

Для текущего уровня важнее уверенно читать код с `&`, `*` и `nil`.

* если функция создаёт новый `map`, результат нужно вернуть

### Задача 16 — необязательный лимит через указатель

Напиши функцию:

```go
func filterByLimit(nums []int, limit *int) []int
```

Требования:

* если `limit == nil`, вернуть исходные значения без фильтрации
* если `limit != nil`, вернуть новый `slice` только с числами `<= *limit`
* исходный `slice` не изменять

Проверь на данных:

```go
nums := []int{3, 10, 2, 8, 5}
limit := 5
```

Вызовы для проверки:

```go
fmt.Println(filterByLimit(nums, nil))
fmt.Println(filterByLimit(nums, &limit))
```

Что здесь тренируется:

* `nil` как отсутствие значения
* безопасное разыменование после проверки
* сбор результата в новый `slice`

### Задача 18 — мини-практика: нормализация чисел

Напиши программу с функциями:

```go
func replaceNegativeWithZero(nums []int)
func appendIfPositive(nums []int, value int) []int
func resetIfNil(p *int) int
```

Требования:

Для `replaceNegativeWithZero`:

* принять `[]int`
* заменить все отрицательные числа на `0`
* не возвращать `slice`, потому что длина не меняется

Для `appendIfPositive`:

* принять `[]int` и число
* если число больше `0`, добавить его через `append`
* вернуть итоговый `slice`

Для `resetIfNil`:

* если указатель равен `nil`, вернуть `0`
* иначе вернуть значение по указателю

Проверь на данных:

```go
nums := []int{3, -1, 5, -7}
extra := 10
```

Что здесь специально тренируется:

* изменение элементов `slice`
* возврат `slice` после `append`
* безопасная работа с `nil`-указателем
* выбор между “изменить на месте” и “вернуть новое значение”

## ⚠️ Важные моменты

* В Go функция получает копии аргументов.
* Обычное присваивание тоже создаёт копию значения.
* `&value` берёт адрес переменной.
* `*int` — тип указателя на `int`.
* `*p` — значение по адресу, который лежит в `p`.
* Через `*p = ...` можно изменить исходное значение.
* Указатель может быть `nil`.
* Перед разыменованием потенциально `nil`-указателя нужна проверка.
* Разыменование `nil` приводит к `panic`.
* Указатели можно сравнивать с `nil` и друг с другом.
* Сравнение указателей сравнивает адреса, а не значения внутри.
* В Go нет обычной арифметики указателей.
* `array` копируется целиком при присваивании и передаче в функцию.
* `slice` копируется как значение, но его элементы могут быть общими.
* Для изменения элементов `slice` обычно достаточно `[]int`.
* После `append` функцию обычно нужно спроектировать так, чтобы она возвращала новый `slice`.
* `map` можно менять внутри функции без `*map`, если меняются ключи и значения.
* Если функция создаёт новый `map`, его нужно вернуть.
* Указатель не должен быть автоматической заменой обычному `return`.
