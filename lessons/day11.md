# 📘 Go | Day 11 — `package`, `import`, структура маленького проекта, экспортируемые и неэкспортируемые имена

---

## 📖 Теория

### 1. Почему после функций пора идти в `package`

На прошлом уроке ты учился выносить логику в маленькие функции.

Это был важный шаг:

* код перестал жить одной “простынёй” внутри `main`
* у отдельных шагов появились понятные имена
* решение стало легче читать и переиспользовать

Но довольно быстро появляется следующая проблема:

* даже если функций много, держать всё в одном файле неудобно
* `main.go` начинает разрастаться
* становится неясно, какие функции относятся к запуску программы, а какие — к самой логике

Именно здесь в Go появляются `package`.

`package` помогает:

* группировать связанный код
* раскладывать программу по файлам и каталогам
* отделять “точку входа” от прикладной логики
* заранее готовить код к ошибкам, тестам и более взрослой структуре проекта

Главная идея урока:

* сначала понять, как Go собирает код по `package`
* потом научиться делить маленькую программу на файлы и пакеты
* затем разобраться, какие имена видны снаружи, а какие должны оставаться внутренними

---

### 2. Каждый `.go`-файл начинается с `package`

В Go у каждого `.go`-файла в начале должна быть строка с именем пакета.

Простейший пример:

```go
package main

import "fmt"

func main() {
	fmt.Println("hello, Go")
}
```

Результат:

```text
hello, Go
```

Что здесь важно:

* `package main` означает, что файл относится к пакету `main`
* имя пакета пишется в самом верху файла
* без строки `package ...` файл не является корректным Go-кодом

На практике это означает:

* Go думает не только категориями “файлов”
* он думает категориями “пакетов”
* файл — это часть пакета, а не самостоятельная программа “сам по себе”

---

### 3. `package main` и `func main`

Для запуска обычной программы нужен пакет `main`.

И внутри него нужна функция:

```go
func main()
```

Пример:

```go
package main

import "fmt"

func normalizeWord(word string) string {
	return word
}

func main() {
	fmt.Println(normalizeWord("go"))
}
```

Результат:

```text
go
```

Почему программа запускается:

* Go находит пакет `main`
* внутри него находит функцию `main`
* именно с неё начинается выполнение программы

Что здесь важно:

* `main` — это точка входа приложения
* остальные функции могут жить рядом, но сами по себе они программу не запускают
* позже мы начнём выносить прикладную логику из `main` в отдельные пакеты

---

### 4. Один `package` может состоять из нескольких файлов

Очень важный практический факт:

* пакет в Go — это не один файл
* пакет может состоять из нескольких файлов в одной папке
* если у файлов одинаковый `package`, Go собирает их вместе

Пример из двух файлов в одной папке.

`main.go`

```go
package main

import "fmt"

func main() {
	raw := []string{" Go ", " SQL ", "php"}
	normalized := normalizeWords(raw)

	fmt.Println(normalized)
	fmt.Println(countLongerThan(normalized, 2))
}
```

`helpers.go`

```go
package main

import "strings"

func normalizeWords(words []string) []string {
	result := make([]string, 0, len(words))

	for _, word := range words {
		clean := strings.ToLower(strings.TrimSpace(word))
		result = append(result, clean)
	}

	return result
}

func countLongerThan(words []string, minLen int) int {
	count := 0

	for _, word := range words {
		if len(word) > minLen {
			count++
		}
	}

	return count
}
```

Запуск:

```text
go run .
```

Результат:

```text
[go sql php]
2
```

Почему это работает:

* оба файла лежат в одной папке
* у обоих файлов указан `package main`
* функция `main` видит `normalizeWords` и `countLongerThan`, хотя они в другом файле
* компилятор собирает эти файлы как один пакет

Это важный мостик от прошлого урока:

* раньше ты делил решение на функции
* теперь ты начинаешь делить решение ещё и на файлы

---

### 5. Для многофайловой программы обычно нужен `go run .`

Когда программа состоит из нескольких файлов одного пакета, удобнее запускать не отдельный файл, а текущую папку целиком:

```text
go run .
```

Если попробовать запустить только `main.go` из прошлого примера:

```text
go run main.go
```

Один из возможных результатов:

```text
./main.go:7:16: undefined: normalizeWords
./main.go:10:14: undefined: countLongerThan
```

Почему так:

* команда `go run main.go` взяла только один файл
* файл `helpers.go` в запуск не попал
* поэтому имена из второго файла оказались “неизвестны”

Практический вывод:

* если программа уже состоит из нескольких файлов в одной папке, чаще всего нужен `go run .`
* та же логика позже будет полезна для `go build .`

Это одна из недостающих микропривычек, без которой тема `package` обычно остаётся “бумажной”.

### 6. `import` подключает другие пакеты

Чтобы использовать функции из стандартной библиотеки или из своих пакетов, нужен `import`.

Пример:

```go
package main

import (
	"fmt"
	"strings"
)

func main() {
	word := " Go "
	clean := strings.ToLower(strings.TrimSpace(word))

	fmt.Println(clean)
}
```

Результат:

```text
go
```

Что здесь происходит:

* `strings.TrimSpace` убирает внешние пробелы
* `strings.ToLower` приводит строку к нижнему регистру
* `fmt.Println` печатает результат

Что важно:

* `import` нужен не “для красоты”, а чтобы Go знал, откуда берутся используемые имена
* запись `strings.ToLower` читается как “функция `ToLower` из пакета `strings`”
* блок импорта в скобках — обычный и удобный стиль, когда импортов несколько

### 7. Импорты принадлежат файлу, а не всему пакету

Это очень частая ловушка у новичков.

Даже если два файла лежат в одном пакете, список импортов у каждого файла свой.

Пример:

`main.go`

```go
package main

import "fmt"

func main() {
	fmt.Println(buildMessage(" Go "))
}
```

`helpers.go`

```go
package main

import "strings"

func buildMessage(word string) string {
	return strings.ToUpper(strings.TrimSpace(word))
}
```

Результат:

```text
GO
```

Почему:

* `fmt` нужен только в `main.go`
* `strings` нужен только в `helpers.go`
* наличие `import "strings"` во втором файле не делает `strings` доступным в первом

Практическое правило:

* каждый файл импортирует только то, что использует сам

И ещё одна важная деталь:

* неиспользуемый импорт в Go — это ошибка компиляции, а не просто предупреждение

Пример:

```go
package main

import (
	"fmt"
	"strconv"
)

func main() {
	fmt.Println("go")
}
```

Один из возможных результатов:

```text
./main.go:5:2: "strconv" imported and not used
```

Это полезно, потому что:

* код не обрастает “мусорными” зависимостями
* список импортов остаётся честным
* проще видеть, от чего реально зависит файл

### 8. Первый собственный пакет: короткий мостик к структуре проекта

Чтобы сделать свой пакет, обычно создают отдельную папку.

Например, структура может быть такой:

```text
wordapp/
  go.mod
  main.go
  words/
    normalize.go
```

`go.mod`

```go
module example.com/wordapp

go 1.25
```

`main.go`

```go
package main

import (
	"fmt"

	"example.com/wordapp/words"
)

func main() {
	fmt.Println(words.Normalize(" Go "))
}
```

`words/normalize.go`

```go
package words

import "strings"

func Normalize(word string) string {
	return strings.ToLower(strings.TrimSpace(word))
}
```

Результат:

```text
go
```

Что здесь важно:

* папка `words` — это отдельный пакет `words`
* `main.go` импортирует его по пути модуля
* функция `Normalize` вызывается как `words.Normalize(...)`

Короткий мостик, который нужен уже сейчас:

* в реальном проекте локальные пакеты живут внутри модуля
* путь модуля хранится в `go.mod`
* подробный разбор `go mod init`, `go mod tidy` и toolchain будет позже
* на этом уроке достаточно понять общую форму: модуль даёт корень, а пакеты живут внутри него

Без этого мостика тема “свои пакеты” получилась бы слишком абстрактной.

### 9. Экспортируемые и неэкспортируемые имена

В Go правило очень простое:

* если имя начинается с заглавной буквы, оно экспортируемое
* если имя начинается со строчной буквы, оно неэкспортируемое

Пример:

`words/normalize.go`

```go
package words

import "strings"

func Normalize(word string) string {
	return clean(word)
}

func clean(word string) string {
	return strings.ToLower(strings.TrimSpace(word))
}
```

Здесь:

* `Normalize` доступна из других пакетов
* `clean` доступна только внутри пакета `words`

Если в `main.go` написать так:

```go
package main

import (
	"fmt"

	"example.com/wordapp/words"
)

func main() {
	fmt.Println(words.clean(" Go "))
}
```

Один из возможных результатов:

```text
undefined: words.clean
```

Почему:

* функция `clean` начинается со строчной буквы
* это внутренняя деталь пакета
* снаружи пакет открывает только экспортируемый API

Практический смысл очень большой:

* экспорт — это не “красивое оформление”
* это способ явно сказать, что пакет обещает наружному коду
* всё остальное лучше по умолчанию держать внутренним

### 10. Неэкспортируемые функции спокойно работают внутри своего пакета

Частая ошибка мышления выглядит так:

* “если функция с маленькой буквы, значит её вообще нельзя вызывать”

Это неверно.

Правильно так:

* функцию с маленькой буквы нельзя вызывать из другого пакета
* внутри своего пакета её можно спокойно использовать

Пример:

```text
wordapp/
  go.mod
  main.go
  words/
    normalize.go
    list.go
```

`words/normalize.go`

```go
package words

import "strings"

func clean(word string) string {
	return strings.ToLower(strings.TrimSpace(word))
}
```

`words/list.go`

```go
package words

func NormalizeAll(items []string) []string {
	result := make([]string, 0, len(items))

	for _, item := range items {
		result = append(result, clean(item))
	}

	return result
}
```

Что здесь важно:

* `clean` не экспортируется
* но `NormalizeAll` может вызывать её, потому что обе функции живут в одном пакете `words`
* это удобный способ прятать внутренние шаги и оставлять наружу только нужные функции

Такой стиль позже особенно полезен:

* для тестируемого кода
* для аккуратного API пакета
* для уменьшения хаоса в проекте

### 11. Как называть пакеты

Хорошее имя пакета обычно:

* короткое
* в нижнем регистре
* без пробелов и дефисов
* по возможности отражает предметную область, а не абстрактную “полезность”

Хорошие примеры:

* `words`
* `users`
* `config`
* `parser`

Плохие или подозрительные примеры:

* `utils`
* `helpers`
* `common`
* `myPackage`
* `string_utils`

Почему `utils` и `helpers` часто плохая идея:

* туда начинает попадать всё подряд
* пакет перестаёт иметь понятную ответственность
* код труднее читать и развивать

Хорошая эвристика:

* если имя пакета не объясняет, за какую область он отвечает, скорее всего имя слабое

Ещё один полезный принцип:

* не делай заикание в стиле `words.WordsNormalize`
* если пакет называется `words`, то функция `Normalize` уже читается нормально: `words.Normalize(...)`

### 12. Структура маленького проекта: `main` оркестрирует, пакет делает работу

Для маленькой программы хороший стартовый стиль часто выглядит так:

* `main` читает входные данные и печатает результат
* прикладная логика живёт в отдельном пакете
* внутри пакета можно делить код на несколько файлов по смыслу

Пример:

```text
wordapp/
  go.mod
  main.go
  words/
    normalize.go
    stats.go
```

`go.mod`

```go
module example.com/wordapp

go 1.25
```

`main.go`

```go
package main

import (
	"fmt"

	"example.com/wordapp/words"
)

func main() {
	raw := []string{" Go ", " SQL ", "go", " php "}
	normalized := words.NormalizeAll(raw)

	fmt.Println(normalized)
	fmt.Println(words.CountUnique(normalized))
}
```

`words/normalize.go`

```go
package words

import "strings"

func NormalizeAll(items []string) []string {
	result := make([]string, 0, len(items))

	for _, item := range items {
		result = append(result, clean(item))
	}

	return result
}

func clean(word string) string {
	return strings.ToLower(strings.TrimSpace(word))
}
```

`words/stats.go`

```go
package words

func CountUnique(items []string) int {
	seen := make(map[string]bool)

	for _, item := range items {
		seen[item] = true
	}

	return len(seen)
}
```

Результат:

```text
[go sql go php]
3
```

Почему это хороший шаг вперёд:

* `main` короткий и читаемый
* логика нормализации и подсчёта живёт не в точке входа, а в пакете
* пакет уже можно воспринимать как маленький API
* такой код намного легче переносить в тесты и расширять дальше

Это как раз тот недостающий мостик, который нужен между функциями и следующими темами про ошибки, тесты и мини-проект.

### 13. Почему этот урок важен перед `error`, `defer` и тестами

Если не разобраться с пакетами сейчас, дальше быстро начнутся проблемы.

Почему:

* в уроке про `error` логично держать обработку ошибок ближе к месту вызова, а не смешивать всё внутри одной гигантской `main`
* в уроке про тесты гораздо удобнее тестировать функции из отдельного пакета, чем код, намертво прилипший к `main`
* в мини-проекте `Phase 1` без структуры по пакетам код почти наверняка превратится в хаос

Поэтому у этого урока не только локальная цель “понять синтаксис `package`”.

Ещё важнее такие навыки:

* видеть границу между входом в программу и прикладной логикой
* по умолчанию не экспортировать лишнее
* группировать код по ответственности, а не по принципу “куда пришлось”
* привыкать запускать проект как пакет, а не как случайный одиночный файл

---

## 🔎 Что особенно важно понять

* `package` в Go важнее отдельного файла: программа собирается пакетами
* `package main` плюс `func main()` нужны для запуска обычного приложения
* несколько файлов с одинаковым `package` в одной папке образуют один пакет
* для многофайловой программы обычно нужен `go run .`, а не `go run main.go`
* импорты задаются отдельно в каждом файле
* заглавная буква открывает имя наружу, строчная оставляет его внутри пакета
* хорошая структура: `main` оркестрирует, пакет делает полезную работу
* короткие и предметные имена пакетов почти всегда лучше, чем `utils/helpers/common`

## ⚠️ Типичные ошибки

### Ошибка 1. Думать, что один файл и есть весь `package`

Из-за этого новичок не понимает, почему функция из соседнего файла “видна” или “не видна”.

Правильно:

* файл — это часть пакета
* компилятор работает на уровне пакета

---

### Ошибка 2. Запускать многофайловую программу через `go run main.go`

Если в проекте есть `main.go` и `helpers.go`, запуск только одного файла часто ломает программу.

Правильная привычка:

* для запуска текущего пакета использовать `go run .`

---

### Ошибка 3. Считать, что импорт из одного файла доступен во всех остальных

Так не работает.

Если `helpers.go` импортирует `strings`, это не значит, что `main.go` тоже может использовать `strings` без своего `import`.

---

### Ошибка 4. Делать почти все имена экспортируемыми “на всякий случай”

Это плохой стартовый стиль.

Если имя не нужно снаружи пакета, лучше оставить его неэкспортируемым.

Почему:

* меньше лишних точек входа
* проще менять внутреннюю реализацию
* API пакета получается аккуратнее

---

### Ошибка 5. Пытаться вызвать `words.clean(...)` из другого пакета

Если имя начинается со строчной буквы, оно не предназначено для внешнего использования.

Это не случайное ограничение, а часть дизайна Go.

---

### Ошибка 6. Складывать несвязанный код в пакет `utils` или `helpers`

Такое имя ничего не говорит о назначении пакета.

Обычно это приводит к тому, что:

* пакет расползается
* логика смешивается
* читать проект становится всё труднее

---

### Ошибка 7. Держать всю прикладную логику в `main`

На маленьком примере это ещё терпимо.

Но как только появляются:

* несколько шагов обработки
* повторное использование функций
* подготовка к тестам

код в `main` начинает мешать, а не помогать.

---

### Ошибка 8. Игнорировать неиспользуемые импорты

В Go это не “мелочь”, а ошибка компиляции.

Это дисциплинирует:

* код остаётся чище
* зависимости остаются очевиднее

---

## 🧠 Полезные паттерны

### 1. Тонкий `main`, толстая прикладная логика вне `main`

Хороший стартовый стиль:

* `main` собирает шаги вместе
* вычисления, нормализация, фильтрация и подсчёты живут в отдельных функциях или пакетах

---

### 2. Неэкспортируемое по умолчанию

Нормальный стартовый вопрос такой:

* “эта функция правда нужна снаружи пакета?”

Если нет, лучше оставить маленькую букву.

---

### 3. Пакет по предметной области, а не по абстрактной “полезности”

Лучше:

* `words`
* `users`
* `config`

Хуже:

* `utils`
* `helpers`
* `common`

### 4. Делить один пакет на несколько файлов по роли

Например:

* `normalize.go`
* `stats.go`
* `main.go`

Так код читается спокойнее, чем один огромный файл на сотни строк.

---

### 5. Пакет как маленький API

Если у пакета снаружи видны только 1-3 понятные функции, это часто хороший знак.

Пример:

* `words.NormalizeAll`
* `words.CountUnique`

А внутренние helper-функции при этом могут оставаться скрытыми.

---

### 6. Подготовка к тестам через чистые функции в отдельном пакете

Даже до официального урока про тесты уже полезно строить код так, чтобы:

* функция принимала данные
* возвращала результат
* не зависела от печати в консоль

Тогда позже её будет легко покрыть тестами.

## 🧠 Сравнение с практикой backend

В backend почти никогда не хочется держать всю логику в одном `main.go`.

Обычно код постепенно делится:

* точка входа приложения
* пакеты с прикладной логикой
* пакеты с работой с данными
* пакеты с HTTP, конфигом, временем, контекстом и так далее

На раннем этапе не нужно искусственно усложнять структуру.

Но уже сейчас полезно привыкать к таким идеям:

* `main` запускает процесс и связывает шаги
* логика обработки данных живёт отдельно
* пакет открывает наружу только нужное
* хорошо названные пакеты делают review и поддержку заметно проще

Именно это потом естественно вырастает в:

* тестируемый код
* аккуратную обработку `error`
* мини-проекты без хаоса

## ⚠️ Важные моменты

* каждый `.go`-файл обязан начинаться со строки `package ...`
* пакет может состоять из нескольких файлов в одной папке
* для многофайловой программы обычно нужен `go run .`
* импорты принадлежат конкретному файлу, а не всему пакету
* неиспользуемые импорты в Go запрещены
* заглавная буква делает имя доступным из других пакетов
* маленькая буква оставляет имя внутренним для пакета
* `main` лучше держать тонким, а полезную логику выносить в функции и пакеты
* хорошие имена пакетов описывают предметную область, а не абстрактную “полезность”
