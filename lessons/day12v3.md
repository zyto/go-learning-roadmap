## 📖 Теория

### 1. Почему после `package` пора идти в `error` и `defer`

На прошлом уроке ты разложил код по функциям, файлам и пакетам.

Это был важный шаг:

* логика перестала жить одной “простынёй” в `main`
* стало проще отделять точку входа от полезной работы
* код начал готовиться к тестам и более взрослой структуре проекта

Но как только программа становится хоть немного реальной, появляется следующая проблема:

* входные данные могут быть плохими
* парсинг может не сработать
* внешний ресурс может не открыться
* из функции иногда нужно выйти раньше, но при этом не забыть про очистку

Именно здесь в Go появляются две ключевые идеи:

* `error` — это обычное значение, которое явно возвращается из функции
* `defer` — это способ гарантированно выполнить очистку при выходе из функции

Главная идея урока:

* научиться видеть не только основной рабочий путь программы, но и путь ошибки
* понять, как функция сообщает о проблеме вызывающему коду
* привыкнуть к `if err != nil { ... }` как к нормальной части программы
* закрепить привычку ставить `defer` рядом с успешным захватом ресурса

Этот урок специально стоит после `day10` и `day11`.

Почему:

* из `day10` у тебя уже есть функции и несколько return values
* из `day11` у тебя уже есть граница между `main` и пакетами
* теперь пора научиться передавать ошибку через эти границы аккуратно и предсказуемо

Перед стартом короткий словарь, чтобы урок не выглядел как набор чужих слов:

* `error` — значение, которое сообщает “что-то пошло не так”
* `err` — обычное имя переменной, в которую часто кладут ошибку
* `nil` — “ничего нет”; для `error` это обычно значит “ошибки нет”
* `defer` — отложить вызов до выхода из текущей функции
* `cleanup` — завершающее действие: закрыть файл, удалить временный файл, освободить ресурс
* `panic` — аварийно прервать обычный ход программы
* `wrapping` — обернуть старую ошибку в новую, добавив контекст
* `context` в этом уроке — не пакет `context`, а дополнительное пояснение “на каком шаге случилась ошибка”

### 2. `error` в Go — это обычное значение

В Go ошибка обычно не “выбрасывается” автоматически, как исключение.

Функция просто возвращает ещё одно значение типа `error`.

Пример:

```go
package main

import (
	"fmt"
	"strconv"
)

func main() {
	n, err := strconv.Atoi("42")

	fmt.Println(n)
	fmt.Println(err == nil)
}
```

Результат:

```text
42
true
```

Почему:

* `strconv.Atoi("42")` успешно преобразовал строку в число
* поэтому число равно `42`
* ошибки нет, значит `err == nil`

Что здесь важно:

* `error` — это часть обычного потока данных программы
* “ошибки нет” в Go обычно означает `err == nil`
* `nil` для `error` — это важный базовый случай, как `0` для `int` или `""` для `string`

### 3. Что такое тип `error` на практическом уровне

Новичку здесь важно не только запомнить слово `error`, но и понять, что это вообще за тип.

На практическом уровне пока достаточно думать так:

* переменная типа `error` хранит либо `nil`, либо некоторое конкретное значение ошибки
* если там `nil`, ошибки нет
* если там не `nil`, значит внутри лежит конкретная ошибка, которую можно напечатать, вернуть из функции или проверить

Формально:

* `error` в Go — это встроенный интерфейсный тип
* интерфейс — это тип, который описывается не полями, а поведением
* в случае `error` это поведение очень маленькое: значение должно уметь вернуть текст ошибки через метод `Error() string`

Слово `метод` пока можно понимать очень просто:

* это функция, которая “принадлежит” значению
* подробно методы будут в отдельном уроке позже

Пока не нужно уметь писать свои типы ошибок вручную.

Для этого урока достаточно понимать:

* `errors.New("empty age")` создаёт конкретное значение ошибки
* `fmt.Errorf(...)` тоже создаёт конкретное значение ошибки
* когда ты пишешь `fmt.Println(err)`, пакет `fmt` печатает текст этой ошибки

Полезная интуиция про “из чего состоит ошибка”:

* ошибка не обязана состоять только из текста
* ошибка из `errors.New(...)` по сути хранит короткое сообщение
* ошибка из `fmt.Errorf("...: %w", err)` хранит и новый текст, и вложенную внутрь старую ошибку

Именно поэтому ниже мы будем различать:

* просто новую ошибку
* и обёрнутую ошибку, внутри которой лежит другая ошибка

### 4. Как создать значение типа `error`: `errors.New`

Самый простой способ создать ошибку — функция `errors.New`.

Она принимает строку и возвращает значение типа `error`.

Пример:

```go
package main

import (
	"errors"
	"fmt"
)

func main() {
	err := errors.New("empty name")

	fmt.Println(err)
	fmt.Println(err == nil)
}
```

Результат:

```text
empty name
false
```

Почему:

* `errors.New("empty name")` создал новое значение ошибки
* это значение не равно `nil`
* при печати выводится текст ошибки

Что здесь важно:

* `errors.New` не печатает ошибку и не завершает программу
* она просто создаёт значение, которое ты потом можешь вернуть, сохранить в переменную или обернуть в другую ошибку

Ещё одна полезная привычка:

* текст ошибки обычно пишут коротко, со строчной буквы и без точки в конце, например `empty name`, а не `Error: Empty Name.`

### 5. Если `err != nil`, результатом пользоваться нельзя “по привычке”

Теперь пример с невалидным вводом:

```go
package main

import (
	"fmt"
	"strconv"
)

func main() {
	n, err := strconv.Atoi("42x")

	fmt.Println(n)
	fmt.Println(err != nil)
	fmt.Println(err)
}
```

Результат:

```text
0
true
strconv.Atoi: parsing "42x": invalid syntax
```

Почему:

* строка `"42x"` не является корректным числом
* `Atoi` не смог выполнить парсинг
* поэтому вернулся `error`
* число `n` в таком случае стало zero value, то есть значением по умолчанию для `int`, а это `0`

Главный практический вывод:

* сначала проверяй `err`
* только потом используй остальные результаты

Неправильная интуиция новичка:

* “раз `n` есть, значит можно им пользоваться”

Правильная привычка:

* при `err != nil` результат почти всегда считается недостоверным

### 6. Идиоматичная сигнатура: данные сначала, `error` последним

Очень частый стиль в Go:

```go
func something(...) (Result, error)
```

Пример:

```go
package main

import (
	"errors"
	"fmt"
	"strconv"
	"strings"
)

func parseAge(text string) (int, error) {
	cleaned := strings.TrimSpace(text)
	if cleaned == "" {
		return 0, errors.New("age is empty")
	}

	age, err := strconv.Atoi(cleaned)
	if err != nil {
		return 0, err
	}

	if age < 0 {
		return 0, errors.New("age must be >= 0")
	}

	return age, nil
}

func main() {
	age, err := parseAge(" 34 ")

	fmt.Println(age)
	fmt.Println(err == nil)
}
```

Результат:

```text
34
true
```

Почему это хороший стиль:

* полезный результат идёт первым
* `error` идёт последним
* при неуспехе функция возвращает zero value результата, то есть значение по умолчанию для этого типа, и ненулевую ошибку
* при успехе возвращает реальный результат и `nil`

Короткий ориентир по выбору сигнатуры:

* `(value, ok)` — когда отсутствие результата нормально и ожидаемо
* `(value, error)` — когда произошла проблема, о которой вызывающий код должен знать

Примеры:

* поиск элемента в `map` или `slice` часто выражается через `ok`
* невалидный ввод, ошибка парсинга, невозможность открыть ресурс — это уже `error`

### 7. `early return`, `guard clause` и `happy path`

Здесь появляются сразу три частых термина.

`early return`:

* ранний выход из функции, когда проблема уже обнаружена и дальше продолжать нет смысла

`guard clause`:

* короткая защитная проверка у входа в проблемный сценарий
* обычно выглядит как `if err != nil { return ... }` или `if input == "" { return ... }`

`happy path`:

* основной рабочий путь программы, когда всё идёт без ошибок

Пример:

```go
package main

import (
	"errors"
	"fmt"
)

func divide(a, b int) (int, error) {
	if b == 0 {
		return 0, errors.New("division by zero")
	}

	return a / b, nil
}

func main() {
	result, err := divide(10, 2)
	fmt.Println(result)
	fmt.Println(err == nil)

	result, err = divide(10, 0)
	fmt.Println(result)
	fmt.Println(err)
}
```

Результат:

```text
5
true
0
division by zero
```

Почему это важно:

* ошибка проверяется сразу у входа в проблемный сценарий
* функция не уходит в лишний `else`
* `happy path` читается легче, потому что не утонул во вложенности

Это прямое продолжение идеи guard clauses из прошлого урока.

Теперь они особенно полезны, потому что с ошибками код без раннего `return` быстро превращается в лес вложенных условий.

### 8. `fmt.Errorf`: что это за функция, что значит `Errorf` и что делает `%w`

`fmt.Errorf` — это функция из пакета `fmt`, которая создаёт новое значение типа `error`.

Имя `Errorf` читается так:

* `Error` — создаём ошибку
* `f` — formatted, то есть “с форматированием”

Это очень похоже на `fmt.Printf`, только:

* `Printf` печатает текст в консоль
* `Errorf` не печатает, а возвращает `error`

Если `Printf` тебе пока не знаком, достаточно запомнить простую версию:

* `fmt.Errorf` принимает шаблон строки с подстановками вроде `%d` и `%w`
* и возвращает готовое значение типа `error`

Короткий фрагмент, а не отдельная программа:

```go
err := fmt.Errorf("port out of range: %d", 70000)
```

Здесь `%d` — это обычное форматирование числа.

Но у `fmt.Errorf` есть особый формат:

* `%w` — wrap, то есть “оберни одну ошибку в другую”

`wrapping`:

* это оборачивание ошибки
* новая ошибка получает свой текст
* и одновременно хранит внутри старую ошибку

Именно поэтому говорят, что у обёрнутой ошибки есть “ошибка внутри”.

Пример:

```go
package main

import (
	"fmt"
	"strconv"
	"strings"
)

func parsePort(text string) (int, error) {
	port, err := strconv.Atoi(strings.TrimSpace(text))
	if err != nil {
		return 0, fmt.Errorf("parse port: %w", err)
	}

	if port < 1 || port > 65535 {
		return 0, fmt.Errorf("parse port: port out of range: %d", port)
	}

	return port, nil
}

func main() {
	_, err := parsePort("abc")
	fmt.Println(err)
}
```

Результат:

```text
parse port: strconv.Atoi: parsing "abc": invalid syntax
```

Что здесь важно:

* `fmt.Errorf("parse port: %w", err)` создал новую ошибку
* в этой новой ошибке есть свой текст `parse port: ...`
* и внутри неё сохранена исходная ошибка `err`

Мысленная схема:

```text
исходная ошибка:
strconv.Atoi: parsing "abc": invalid syntax

новая ошибка после wrapping:
parse port: [исходная ошибка внутри]

то, что видит fmt.Println:
parse port: strconv.Atoi: parsing "abc": invalid syntax
```

Практический смысл:

* нижний уровень знает “что именно сломалось”
* верхний уровень знает “на каком шаге это случилось”

Если исходной ошибки нет, а ты просто формируешь новую, `%w` не нужен.

Например, здесь:

```go
fmt.Errorf("parse port: port out of range: %d", port)
```

мы не оборачиваем чужую ошибку, а создаём новую ошибку с текстом.

### 9. `errors.Is`, wrapping и `sentinel error`

Если ошибка была обёрнута через `%w`, её всё ещё можно проверить через `errors.Is`.

Здесь появляется ещё один термин:

`sentinel error`:

* это заранее объявленная ошибка-переменная
* она используется как стабильная метка конкретной проблемы
* код потом может спрашивать: “это именно ошибка `ErrEmptyName` или нет?”

Пример:

```go
package main

import (
	"errors"
	"fmt"
	"strings"
)

var ErrEmptyName = errors.New("empty name")

func normalizeName(name string) (string, error) {
	clean := strings.TrimSpace(name)
	if clean == "" {
		return "", ErrEmptyName
	}

	return strings.ToLower(clean), nil
}

func loadPrimaryName(raw string) (string, error) {
	name, err := normalizeName(raw)
	if err != nil {
		return "", fmt.Errorf("load primary name: %w", err)
	}

	return name, nil
}

func main() {
	_, err := loadPrimaryName("   ")

	fmt.Println(err)
	fmt.Println(errors.Is(err, ErrEmptyName))
}
```

Результат:

```text
load primary name: empty name
true
```

Почему это полезно:

* верхний слой получил уже обёрнутую ошибку
* но через `errors.Is` всё ещё можно понять корневую причину
* это особенно удобно для `sentinel error`, которая означает конкретный ожидаемый тип проблемы

Мысленная схема цепочки:

```text
ErrEmptyName
    ↑
load primary name: %w
```

`errors.Is(err, ErrEmptyName)` идёт по этой цепочке и спрашивает:

* текущая ошибка — это `ErrEmptyName`?
* если нет, есть ли внутри ещё одна ошибка?
* если есть, проверить её тоже

Поэтому результат `true`, хотя снаружи текст ошибки уже выглядит как `load primary name: empty name`.

Здесь важно не спутать два сценария:

* “показать человеку понятный текст” — для этого нужно хорошее сообщение об ошибке
* “понять в коде, что именно произошло” — для этого нужен `errors.Is`

Именно поэтому при wrapping лучше использовать `%w`, а не просто `%v`.

### 10. Короткий мостик к `errors.As`

Ещё одна важная функция из пакета `errors` — это `errors.As`.

Она отвечает на другой вопрос:

* не “это ли именно такая ошибка?”
* а “можно ли трактовать ошибку как такой-то конкретный тип?”

Сегодня в неё глубоко не уходим.

Причина простая:

* `errors.As` быстро упирается в конкретные типы ошибок
* там очень скоро появляются детали, которые тесно связаны с указателями и устройством значений
* следующий урок как раз подводит нас ближе к `value semantics`, то есть к тому, как значения копируются и передаются, и к указателям

На сегодня достаточно помнить:

* `errors.Is` — проверка причины
* `errors.As` — извлечение ошибки как конкретного типа

### 11. Что такое `panic`

`panic` — это встроенная функция Go для аварийного прерывания обычного хода программы.

Когда вызывается `panic(...)`:

* текущая функция не продолжает нормальное выполнение
* Go начинает подниматься вверх по стеку вызовов
* если никто не перехватил `panic`, программа завершается и печатает сообщение об ошибке и трассировку стека

`трассировка стека` или `stack trace`:

* это список вызовов функций, по которому видно, где программа упала

Пример:

```go
package main

import "fmt"

func main() {
	fmt.Println("before panic")
	panic("bad input")
	fmt.Println("after panic")
}
```

Один из возможных результатов:

```text
before panic
panic: bad input
```

Почему:

* строка `before panic` успела выполниться
* затем вызвался `panic`
* строка `after panic` уже не выполнилась
* после этого Go напечатал сообщение `panic: bad input`

Дальше Go обычно печатает ещё и трассировку стека.

Её точный вид зависит от файла, строк и среды запуска.

Механизм перехвата `panic` через `recover` существует, но для этого урока его знать не нужно.

После объяснения `defer` мы отдельно вернёмся к важной детали: даже при `panic` отложенные вызовы `defer` всё равно успевают выполниться.

### 12. `panic` не заменяет нормальную обработку ошибок

Для ожидаемых прикладных проблем в Go обычно используют `error`, а не `panic`.

Фрагмент-антипаттерн:

Это именно короткий фрагмент, а не полноценная запускаемая программа:

```go
func parseAge(text string) int {
	age, err := strconv.Atoi(text)
	if err != nil {
		panic(err)
	}

	return age
}
```

Почему это плохой стартовый стиль:

* пользователь ввёл плохое значение — это обычная рабочая ситуация
* программа не обязана падать целиком
* вызывающий код теряет возможность аккуратно решить, что делать дальше

Правильнее:

* вернуть `error`
* дать вызывающему коду выбрать реакцию

В ранних темах полезно закрепить простое правило:

* ожидаемая проблема ввода, парсинга, валидации, открытия ресурса — это `error`
* `panic` — не основной инструмент прикладной логики

### 13. Что такое `defer` и когда он выполняется

`defer` — это ключевое слово Go.

Оно не вызывает функцию прямо сейчас.

Оно регистрирует вызов на момент, когда текущая функция будет завершаться.

`cleanup`:

* это завершающее действие, которое нужно выполнить перед выходом
* типичные примеры: `Close`, `Remove`, `Unlock`, печать финального сообщения

Базовая форма выглядит так:

```go
defer someFunc()
```

Это значит:

* сейчас вызов не выполняется
* Go “запоминает” его
* когда текущая функция завершится, вызов будет выполнен

Это особенно удобно для cleanup-действий.

Пример:

```go
package main

import (
	"errors"
	"fmt"
	"strings"
)

func process(text string) error {
	fmt.Println("begin")
	defer fmt.Println("cleanup")

	if strings.TrimSpace(text) == "" {
		return errors.New("empty input")
	}

	fmt.Println("processing")
	return nil
}

func main() {
	_ = process("   ")
}
```

Результат:

```text
begin
cleanup
```

Почему:

* функция начала работу и сразу зарегистрировала cleanup через `defer`
* затем сработал ранний `return`
* перед окончательным выходом из функции Go выполнил отложенный вызов

Это одна из главных причин, почему `defer` так полезен рядом с ошибками:

* у функции может быть много точек выхода
* cleanup хочется написать один раз

---

### 14. Как `panic` связан с `defer`

Теперь можно вернуться к детали из раздела про `panic`.

Даже если функция завершается не обычным `return`, а через `panic`, уже зарегистрированные `defer` всё равно выполняются.

Пример:

```go
package main

import "fmt"

func main() {
	defer fmt.Println("deferred cleanup")

	fmt.Println("before panic")
	panic("bad input")
}
```

Один из возможных результатов:

```text
before panic
deferred cleanup
panic: bad input
```

Почему:

* `defer` был зарегистрирован до `panic`
* затем программа дошла до `panic`
* обычный ход программы прервался
* но перед завершением Go выполнил отложенный вызов

Это не значит, что `panic` теперь нужно использовать для обычных ошибок.

Это значит только одно:

* `defer` — надёжный механизм cleanup даже при аварийном выходе

---

### 15. Несколько `defer` выполняются в обратном порядке

Это поведение называют `LIFO`:

* `last in, first out`
* по-русски: “последним поставили, первым выполнили”

Пример:

```go
package main

import "fmt"

func main() {
	fmt.Println("start")
	defer fmt.Println("first defer")
	defer fmt.Println("second defer")
	fmt.Println("finish body")
}
```

Результат:

```text
start
finish body
second defer
first defer
```

Почему:

* второй `defer` был зарегистрирован позже
* значит, он выполнится раньше первого

Это не мелочь.

На реальной очистке ресурсов порядок иногда важен.

---

### 16. Аргументы `defer` вычисляются сразу

Очень частая ловушка:

* сам вызов откладывается
* но аргументы для него вычисляются в момент записи `defer`

Пример:

```go
package main

import "fmt"

func main() {
	value := 10
	defer fmt.Println("deferred:", value)

	value = 20
	fmt.Println("now:", value)
}
```

Результат:

```text
now: 20
deferred: 10
```

Почему:

* значение `value` для отложенного `Println` было захвачено в момент строки `defer`
* позже переменная изменилась, но аргумент уже был вычислен

Это важно помнить, чтобы не делать ложных выводов о “магии” `defer`.

---

### 17. Паттерн “захватил ресурс -> проверил ошибку -> сразу поставил `defer`”

Именно так `defer` чаще всего выглядит в реальном коде.

Пример:

```go
package main

import (
	"fmt"
	"os"
)

func main() {
	file, err := os.Create("temp.txt")
	if err != nil {
		fmt.Println("create error:", err)
		return
	}

	defer os.Remove("temp.txt")
	defer file.Close()

	fmt.Println("temp.txt created")
}
```

Результат:

```text
temp.txt created
```

Почему это хороший шаблон:

* сначала ресурс получен
* сразу проверено, что ошибки нет
* сразу рядом поставлен cleanup
* потом уже идёт остальная логика

Здесь есть ещё один полезный нюанс:

* `defer os.Remove("temp.txt")` записан раньше
* `defer file.Close()` записан позже
* поэтому при выходе сначала выполнится `file.Close()`, а уже потом `os.Remove("temp.txt")`

Подробная работа с файлами будет позже.

Сегодня важно понять именно шаблон:

* не откладывай cleanup “на потом внизу функции”
* ставь `defer` рядом с успешным получением ресурса

---

### 18. Ошибку обычно возвращает пакет, а решает её уже `main` или вызывающий код

Это важный мостик от `day11`.

Хороший стартовый стиль выглядит так:

* пакет делает полезную работу
* пакет возвращает результат или `error`
* `main` решает, что печатать, как завершаться и что считать пользовательским сообщением

Пример структуры:

```text
day12app/
  go.mod
  main.go
  words/
    normalize.go
```

`go.mod`

```go
module example.com/day12app

go 1.25
```

`main.go`

```go
package main

import (
	"fmt"

	"example.com/day12app/words"
)

func main() {
	word, err := words.Normalize("   ")
	if err != nil {
		fmt.Println("error:", err)
		return
	}

	fmt.Println(word)
}
```

`words/normalize.go`

```go
package words

import (
	"errors"
	"strings"
)

var ErrEmptyWord = errors.New("empty word")

func Normalize(word string) (string, error) {
	clean := strings.TrimSpace(word)
	if clean == "" {
		return "", ErrEmptyWord
	}

	return strings.ToLower(clean), nil
}
```

Запуск:

```text
go run .
```

Результат:

```text
error: empty word
```

Почему это важно:

* пакет `words` не печатает и не падает самовольно
* он честно возвращает ошибку наружу
* `main` остаётся точкой, где принимается решение о реакции

Это особенно полезно для будущих тем:

* тесты проще писать против функции, которая возвращает `error`
* мини-проект проще поддерживать, если ошибки не размазаны по `fmt.Println` внутри helper-функций, то есть вспомогательных функций

---

### 19. Почему этот урок важен перед следующими шагами

Если не закрепить `error` и `defer` сейчас, дальше начнутся проблемы почти во всех практических темах.

Почему:

* в `day13` и дальше станет важнее понимать, что именно передаётся как значение и где появляется `nil`
* в теме тестов удобнее всего проверять функции, которые возвращают результат и `error`
* в уроках про файлы, JSON, `context`, `CLI` (командные утилиты) и backend-код ошибки и `cleanup` будут встречаться постоянно
* без `wrapping` сообщения быстро становятся слишком плоскими и непонятными
* без `defer` cleanup легко забывается на одном из ранних `return`

То есть этот урок нужен не только сам по себе.

Он создаёт базу для:

* надёжного прикладного кода
* понятного поведения на ошибках
* более взрослых привычек до начала мини-проекта `Phase 1`

---

## 🔎 Что особенно важно понять

После этого урока ты должен уверенно различать:

* `error` как обычное возвращаемое значение, а не “скрытую магию”
* `nil` как состояние “ошибки нет”
* `(value, ok)` и `(value, error)` как два разных смысловых сигнала
* `errors.New` для простых ошибок и `fmt.Errorf(... %w ...)` для `wrapping`, то есть оборачивания ошибки с контекстом
* `errors.Is` как способ проверить корневую причину после wrapping
* разницу между `error` и `panic` для обычных прикладных проблем
* `defer` как гарантированный `cleanup`, то есть завершающее действие при любом выходе из функции
* связь `panic` и `defer`: аварийный выход не отменяет уже зарегистрированные отложенные вызовы
* LIFO-порядок `defer`, то есть “последним добавили, первым выполнили”
* правило: захватил ресурс, проверил ошибку, сразу поставил `defer`

Если эти вещи станут ясны, дальше темы про указатели, тесты, файлы и мини-проекты пойдут намного спокойнее.

## ⚠️ Типичные ошибки

### Ошибка 1. Думать, что `error` в Go работает как исключение

Нет.

В обычном Go-коде ошибка:

* возвращается явно
* проверяется явно
* не исчезает сама по себе

Если функцию с `error` вызвать и не обработать результат, проблема не “решится автоматически”.

---

### Ошибка 2. Пользоваться результатом до проверки `err`

Это одна из самых вредных ранних привычек.

Если функция вернула:

```go
value, err := something()
```

то сначала нужно понять, что произошло с `err`.

Только потом решать, можно ли доверять `value`.

---

### Ошибка 3. Путать `(value, ok)` и `(value, error)`

Не каждое отсутствие значения — это ошибка.

Например:

* “элемент не найден” часто нормально выражается через `ok == false`
* “строка не парсится в число” — это уже `error`

Если путать эти сценарии, внешний API функции, то есть её контракт для вызывающего кода, становится размытым.

---

### Ошибка 4. Оборачивать ошибку через `%v`, а потом ждать, что `errors.Is` сработает

Если сделать так:

```go
fmt.Errorf("parse port: %v", err)
```

то получится только текст.

Исходная ошибка не будет сохранена как часть цепочки wrapping.

Для сохранения причины нужен `%w`.

---

### Ошибка 5. Сравнивать обёрнутую ошибку через `==`

После wrapping лучше думать не в стиле:

```go
err == ErrEmptyName
```

а в стиле:

```go
errors.Is(err, ErrEmptyName)
```

Именно `errors.Is` умеет пройти по цепочке обёртываний.

---

### Ошибка 6. Печатать ошибку глубоко внутри helper-функции и не возвращать её наружу

Плохой стартовый стиль выглядит так:

* helper-функция, то есть вспомогательная функция, сама делает `fmt.Println(err)`
* сама решает, что пользователю показывать
* и часто ещё прячет проблему от вызывающего кода

Гораздо лучше:

* вернуть `error`
* дать `main` или вызывающему слою решить, что делать

---

### Ошибка 7. Использовать `panic` для обычной валидации и парсинга

Пустая строка, плохое число, неверный порт, неудачный ввод пользователя:

* это не повод сразу ронять программу
* это обычные рабочие проблемы

Для таких случаев почти всегда нужен `error`.

---

### Ошибка 8. Ставить `defer file.Close()` до проверки `err`

Опасный шаблон:

```go
file, err := os.Open("data.txt")
defer file.Close()
if err != nil {
	return
}
```

Почему это плохо:

* если открытие не удалось, `file` может быть неготов к использованию
* cleanup регистрируется слишком рано

Правильнее:

* сначала проверить `err`
* потом ставить `defer`

---

### Ошибка 9. Думать, что `defer` выполняется сразу

Нет.

Сразу выполняется только регистрация отложенного вызова.

Сам вызов произойдёт при выходе из текущей функции.

Из-за этой путаницы легко неправильно понимать порядок операций и значения переменных.

---

### Ошибка 10. Бездумно ставить `defer` внутри большого цикла

`defer` отложит cleanup до конца функции, а не до конца итерации.

Это значит:

* cleanup может накопиться
* ресурс будет жить дольше, чем ты ожидал

Для этого урока достаточно запомнить риск.

Полноценнее к таким ситуациям вернёмся на более практичных задачах с файлами, `io` и более поздним кодом.

---

## 🧠 Полезные паттерны

### 1. Guard clause на ошибке

То есть защитная проверка с ранним выходом.

Базовый шаблон:

```go
if err != nil {
	return zeroValue, err
}
```

Он делает код короче и честнее.

---

### 2. Данные сначала, `error` последним

Хорошая сигнатура обычно выглядит так:

```go
func ParseSomething(input string) (Result, error)
```

Это читается привычно и ожидаемо для любого Go-разработчика.

---

### 3. Короткий контекст на границе слоя через `fmt.Errorf`

Хороший стиль:

* низкий слой возвращает исходную причину
* верхний слой добавляет свой контекст

Например:

```go
return 0, fmt.Errorf("load config port: %w", err)
```

---

### 4. Sentinel error + `errors.Is`

Если важна именно категория проблемы, а не только текст, удобно завести `sentinel error`, то есть заранее объявленную ошибку-переменную:

```go
var ErrEmptyName = errors.New("empty name")
```

И проверять её так:

```go
errors.Is(err, ErrEmptyName)
```

---

### 5. Захватил ресурс, проверил ошибку, сразу поставил `defer`

Не так:

* получить ресурс
* долго писать логику
* в конце вспомнить про `Close`

А так:

* получил ресурс
* убедился, что ошибки нет
* сразу рядом написал cleanup

---

### 6. Пакет возвращает `error`, край программы решает реакцию

Хороший стартовый стиль:

* пакет не печатает без необходимости
* пакет не делает `panic` на обычной проблеме
* `main`, `handler` (обработчик, например HTTP-обработчик) или вызывающая функция решают, что показать наружу

---

### 7. `(value, ok)` для нормального отсутствия, `error` для проблемы

Это полезная развилка после прошлого урока:

* не найдено в коллекции — часто `ok`
* сломан ввод, парсинг, внешний шаг — это уже `error`

Так API функции, то есть внешний контракт функции, становится яснее.

---

## 🧠 Сравнение с практикой backend

В backend тема `error` и `defer` встречается буквально везде:

* чтение конфига и env vars
* парсинг JSON и query-параметров
* работа с файлами
* HTTP-слой
* база данных
* Redis
* внешние API

Почему это настолько важно:

* `happy path`, то есть сценарий без ошибок, почти никогда не единственный
* ошибка должна не просто существовать, а быть понятной по контексту
* `cleanup`, то есть завершающие действия, нельзя держать “в голове” — он должен быть привязан к коду

Очень частый реальный стиль:

* функция сервиса вызывает функцию репозитория
* репозиторий возвращает ошибку
* сервис добавляет контекст
* верхний слой решает, что показать клиенту или записать в лог

А `defer` регулярно используется для:

* `Close`
* cleanup временных ресурсов
* простых завершающих действий при выходе из функции

Что здесь специально тренируется:

* `defer` ставится сразу после успешного получения ресурса
* порядок нескольких cleanup-действий имеет значение

### Задача 12 — мини-практика на пакет, wrapping и `errors.Is`

Собери маленький проект со структурой:

```text
day12mini/
  go.mod
  main.go
  numbers/
    parse.go
    sum.go
```

Предлагаемый модуль:

```go
module example.com/day12mini
```

Сделай в пакете `numbers`:

```go
var ErrEmptyNumber = errors.New("empty number")

func ParsePositive(text string) (int, error)
func SumFromStrings(items []string) (int, error)
```

Требования:

Для `ParsePositive`:

* убирать пробелы
* возвращать `ErrEmptyNumber`, если строка пустая
* парсить число через `strconv.Atoi`
* возвращать ошибку через `fmt.Errorf(...)`, если число меньше или равно `0`

Для `SumFromStrings`:

* пройти по `[]string`
* для каждого элемента вызвать `ParsePositive`
* при ошибке добавить контекст с индексом, например через `fmt.Errorf("item %d: %w", i, err)`
* при успехе вернуть сумму

В `main`:

* вызови `numbers.SumFromStrings(...)`
* если ошибка связана с `numbers.ErrEmptyNumber`, определи это через `errors.Is`
* выведи либо сумму, либо понятное сообщение об ошибке

Что здесь специально тренируется:

* граница между `main` и пакетом
* возврат ошибок наружу вместо скрытого `fmt.Println`
* wrapping, то есть оборачивание ошибки с контекстом
* проверка причины через `errors.Is`

## ⚠️ Важные моменты

* `error` в Go почти всегда возвращается явно из функции
* `nil` для `error` означает, что ошибки нет
* сначала проверяй `err`, потом используй остальные результаты
* `errors.New` подходит для простых ошибок, `fmt.Errorf(... %w ...)` — для wrapping, то есть оборачивания ошибки с контекстом
* `errors.Is` нужен, чтобы узнавать причину ошибки даже после wrapping
* `panic` не должен быть заменой обычной валидации и парсинга
* `defer` выполняется при выходе из функции, в том числе на раннем `return`
* уже зарегистрированные `defer` выполняются и при `panic`
* несколько `defer` работают по принципу `LIFO`, то есть “последним добавили, первым выполнили”
* аргументы `defer` вычисляются в момент записи строки `defer`
* хороший шаблон: получить ресурс, проверить ошибку, сразу поставить cleanup, то есть завершающее действие, через `defer`
* пакет обычно возвращает `error`, а `main` или внешний слой решают, как реагировать
