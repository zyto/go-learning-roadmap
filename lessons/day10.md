## 📖 Теория

### 1. Почему после коллекций и stdlib пора идти в функции

В прошлых уроках ты уже делал полезные вещи:

* проходил `slice`
* строил `map`
* нормализовал строки
* использовал функции из stdlib

Но если весь код писать прямо в `main`, довольно быстро появляются проблемы:

* решение становится длинным
* отдельные шаги трудно переиспользовать
* сложнее читать логику сверху вниз
* неудобно потом тестировать отдельные части

Функция нужна не только для “не повторять код”.

Ещё важнее другое:

* функция даёт имя куску логики
* функция задаёт понятный вход и выход
* функция помогает разбить решение на маленькие шаги

На этом уроке главная идея такая:

* сначала научиться уверенно оформлять алгоритмы в функции
* потом использовать это как мостик к пакетам, тестам и ошибкам

### 2. Базовая форма функции

Самая простая функция в Go выглядит так:

```go
func sumTwo(a int, b int) int {
	return a + b
}
```

Что здесь означает каждая часть:

* `func` — объявление функции
* `sumTwo` — имя функции
* `a int, b int` — параметры
* `int` после скобок — тип возвращаемого результата
* `return` — вернуть значение и завершить функцию

Пример использования:

```go
result := sumTwo(10, 20)
fmt.Println(result)
```

Результат:

```text
30
```

Что здесь важно:

* сигнатура функции — это её контракт
* по сигнатуре должно быть понятно, что функция принимает и что отдаёт
* хорошее имя функции обычно делает код заметно понятнее

### 3. Параметры и аргументы

Параметры и аргументы — это не одно и то же.

Пример:

```go
func isBetween(n int, left int, right int) bool {
	return n >= left && n <= right
}

ok := isBetween(7, 1, 10)
```

Здесь:

* `n`, `left`, `right` — параметры функции
* `7`, `1`, `10` — аргументы при вызове

Если несколько параметров одного типа идут подряд, тип можно писать короче:

```go
func isBetween(n, left, right int) bool {
	return n >= left && n <= right
}
```

Это обычный и удобный стиль Go.

Что важно на практике:

* параметры должны отражать смысл задачи
* лучше `target`, `limit`, `prefix`, чем размытые `x`, `y`, `data`
* если сигнатура выглядит перегруженной, это часто сигнал, что функцию стоит упростить

### 4. Функция может возвращать одно значение

Пример:

```go
func isEven(n int) bool {
	return n%2 == 0
}
```

Использование:

```go
fmt.Println(isEven(10))
fmt.Println(isEven(7))
```

Результат:

```text
true
false
```

Такие функции особенно удобны, когда:

* нужно вынести маленькую проверку
* хочется дать имя условию
* один и тот же кусок логики повторяется в нескольких местах

Пример с алгоритмом:

```go
func sumSlice(nums []int) int {
	sum := 0

	for _, num := range nums {
		sum += num
	}

	return sum
}
```

Что здесь важно:

* логика суммирования теперь вынесена из `main`
* имя `sumSlice` сразу объясняет намерение
* эту функцию потом легко переиспользовать и тестировать

### 5. Функция может возвращать несколько значений

Это одна из очень характерных особенностей Go.

Пример:

```go
func minMax(nums []int) (int, int) {
	min := nums[0]
	max := nums[0]

	for _, num := range nums[1:] {
		if num < min {
			min = num
		}

		if num > max {
			max = num
		}
	}

	return min, max
}
```

Использование:

```go
nums := []int{7, 3, 10, 2, 8}
min, max := minMax(nums)

fmt.Println(min, max)
```

Результат:

```text
2 10
```

Что здесь важно:

* функция может вернуть сразу несколько результатов
* это часто делает код проще, чем искусственное упаковывание всего в одну переменную
* в Go такой стиль очень нормален и встречается постоянно

Но здесь есть важный нюанс:

* эта версия функции предполагает, что `slice` не пустой

Если пустой `slice` возможен, сигнатуру обычно нужно сделать честнее.

### 6. Частый паттерн: результат + признак успеха

Когда функция может “не найти” значение или не суметь корректно сработать для части входных данных, полезно вернуть дополнительный `bool`.

Пример:

```go
func findIndex(nums []int, target int) (int, bool) {
	for i, num := range nums {
		if num == target {
			return i, true
		}
	}

	return -1, false
}
```

Использование:

```go
nums := []int{10, 20, 30}

idx, ok := findIndex(nums, 20)
fmt.Println(idx, ok)

idx, ok = findIndex(nums, 99)
fmt.Println(idx, ok)
```

Результат:

```text
1 true
-1 false
```

Почему это хороший паттерн:

* сигнатура честно говорит, что поиск может не сработать
* вызывающий код не должен гадать по “магическому” значению
* это естественный мостик к будущему стилю `value, error`

Похожим образом можно сделать и `minMax` безопаснее:

```go
func minMax(nums []int) (int, int, bool) {
	if len(nums) == 0 {
		return 0, 0, false
	}

	min := nums[0]
	max := nums[0]

	for _, num := range nums[1:] {
		if num < min {
			min = num
		}

		if num > max {
			max = num
		}
	}

	return min, max, true
}
```

### 7. `return` завершает функцию сразу

Это простая, но очень важная идея.

Пример:

```go
func firstPositive(nums []int) (int, bool) {
	for _, num := range nums {
		if num > 0 {
			return num, true
		}
	}

	return 0, false
}
```

Что здесь важно:

* как только выполняется `return`, функция заканчивается
* это очень удобно для поиска, проверок и guard clauses
* позже этот стиль особенно пригодится при работе с `error`

Guard clause — это ранний выход из функции, когда дальнейшая работа не нужна.

Пример:

```go
func average(nums []int) (float64, bool) {
	if len(nums) == 0 {
		return 0, false
	}

	sum := 0

	for _, num := range nums {
		sum += num
	}

	return float64(sum) / float64(len(nums)), true
}
```

Такой стиль обычно читается легче, чем лишняя вложенность.

### 8. `scope`: где переменная видна, а где нет

У каждой переменной есть область видимости.

Пример:

```go
func demo() {
	total := 0

	for i := 1; i <= 3; i++ {
		step := i * 10
		total += step
	}

	fmt.Println(total)
}
```

Что здесь важно:

* `total` видна во всей функции `demo`
* `i` видна только внутри цикла `for`
* `step` тоже видна только внутри цикла

Такой код не скомпилируется:

```go
func demo() {
	for i := 1; i <= 3; i++ {
		step := i * 10
	}

	fmt.Println(step)
}
```

Почему:

* `step` была объявлена внутри блока цикла
* за пределами этого блока переменная уже не существует

Практический вывод:

* не тащи переменные шире, чем реально нужно
* держи область видимости узкой, но понятной
* внимательно следи за `:=`, чтобы случайно не создать новую переменную внутри блока

### 9. Маленькие функции лучше большой “простыни”

Сравним подходы.

Большой кусок логики в одном месте:

```go
func main() {
	raw := []string{" Go ", "php", " SQL "}
	result := make([]string, 0, len(raw))

	for _, word := range raw {
		word = strings.TrimSpace(word)
		word = strings.ToLower(word)

		if word != "" {
			result = append(result, word)
		}
	}

	fmt.Println(result)
}
```

Это рабочий код, но он быстро начнёт разрастаться.

Более удобный вариант:

```go
func normalizeWord(word string) string {
	word = strings.TrimSpace(word)
	word = strings.ToLower(word)
	return word
}

func filterNonEmpty(words []string) []string {
	result := make([]string, 0, len(words))

	for _, word := range words {
		if word != "" {
			result = append(result, word)
		}
	}

	return result
}

func normalizeWords(words []string) []string {
	result := make([]string, 0, len(words))

	for _, word := range words {
		result = append(result, normalizeWord(word))
	}

	return filterNonEmpty(result)
}
```

Почему второй вариант лучше:

* каждый шаг получил имя
* код легче читать сверху вниз
* отдельные части проще тестировать
* следующий урок про `package` станет естественнее

### 10. Функции для алгоритмов на `slice`

После предыдущих уроков уже естественно оформлять типовые алгоритмы как функции.

Пример фильтрации:

```go
func filterEven(nums []int) []int {
	result := make([]int, 0, len(nums))

	for _, num := range nums {
		if num%2 == 0 {
			result = append(result, num)
		}
	}

	return result
}
```

Пример поиска:

```go
func findFirstLongerThan(words []string, minLen int) (string, bool) {
	for _, word := range words {
		if len(word) > minLen {
			return word, true
		}
	}

	return "", false
}
```

На текущем этапе полезно держаться простого правила:

* если функция строит новый результат, обычно лучше вернуть новый `slice`
* если функция что-то ищет, часто уместен паттерн `(value, ok)`
* если функция делает сразу много разных вещей, скорее всего её пора разбить

### 11. Variadic functions

Иногда функция должна принимать произвольное количество аргументов одного типа.

Для этого в Go есть variadic functions.

Пример:

```go
func sumAll(nums ...int) int {
	sum := 0

	for _, num := range nums {
		sum += num
	}

	return sum
}
```

Использование:

```go
fmt.Println(sumAll(10, 20))
fmt.Println(sumAll(1, 2, 3, 4))
fmt.Println(sumAll())
```

Результат:

```text
30
10
0
```

Что здесь важно:

* запись `nums ...int` означает “любое количество `int`”
* внутри функции `nums` ведёт себя как обычный `[]int`
* variadic-функция особенно удобна, когда у вызывающего кода есть отдельные значения, а не заранее собранный `slice`

---

### 12. Передача `slice` в variadic-функцию через `...`

В `Day 09` ты уже видел запись `b...` в `append`.

Теперь можно понять её полностью.

Если есть variadic-функция:

```go
func sumAll(nums ...int) int {
	sum := 0

	for _, num := range nums {
		sum += num
	}

	return sum
}
```

и есть готовый `slice`:

```go
values := []int{5, 10, 15}
```

то вызвать функцию можно так:

```go
fmt.Println(sumAll(values...))
```

Почему нужен `...`:

* `sumAll` ждёт не один аргумент типа `[]int`
* она ждёт набор отдельных значений типа `int`
* запись `values...` “раскрывает” `slice` в набор аргументов

Без `...` будет ошибка компиляции:

```go
sumAll(values) // ошибка
```

Это важный практический мостик:

* `append(dst, src...)`
* `sumAll(values...)`

Обе записи опираются на одну и ту же идею раскрытия `slice`.

### 13. Как выбирать сигнатуру функции

На текущем этапе полезно запомнить несколько типовых форм.

Если задача всегда работает с готовым `slice`:

```go
func sumSlice(nums []int) int
```

Если задача ищет значение и поиск может не сработать:

```go
func findIndex(nums []int, target int) (int, bool)
```

Если задача строит новый набор данных:

```go
func filterEven(nums []int) []int
```

Если функция удобна именно для произвольного числа аргументов:

```go
func sumAll(nums ...int) int
```

Хорошая сигнатура обычно отвечает на 3 вопроса:

* что функция принимает
* что функция возвращает
* что может пойти “не по обычному сценарию”

Это уже подготовка к следующим урокам:

* в `Day 11` функции начнут жить в отдельных файлах и пакетах
* в `Day 12` вместо `ok` часто появится `error`

## 🔎 Что особенно важно понять

* функция — это не просто “кусок вынесенного кода”, а именованный шаг решения
* сигнатура функции — это контракт, который должен быть честным и понятным
* несколько возвращаемых значений в Go — нормальный и очень частый стиль
* паттерн `(value, ok)` полезен там, где результат может отсутствовать
* `return` сразу завершает функцию
* variadic-параметр внутри функции работает как `slice`
* запись `values...` нужна, чтобы передать готовый `slice` в variadic-функцию
* маленькие функции упрощают чтение, переиспользование и будущие тесты

## ⚠️ Типичные ошибки

### Ошибка 1. Оставлять всю логику в `main`

Если решение уже содержит несколько шагов, но всё ещё лежит в одном длинном блоке, код быстро становится трудно читать и менять.

Признак проблемы:

* в `main` уже и чтение данных, и нормализация, и фильтрация, и поиск, и вывод

---

### Ошибка 2. Давать функции размытое имя

Плохие имена вроде `process`, `handle`, `doStuff` почти ничего не объясняют.

Лучше:

* `sumSlice`
* `findIndex`
* `normalizeWord`
* `filterNonEmpty`

Имя должно помогать понять намерение без чтения всей реализации.

---

### Ошибка 3. Прятать “не найдено” внутри zero value

Например, функция возвращает `0`, когда элемент не найден.

Проблема:

* `0` может быть и реальным корректным результатом
* вызывающему коду приходится гадать, что произошло

Поэтому для таких случаев часто лучше:

```go
func findIndex(nums []int, target int) (int, bool)
```

---

### Ошибка 4. Не учитывать пустой `slice`

Функция вроде:

```go
func minMax(nums []int) (int, int) {
	min := nums[0]
	max := nums[0]
	return min, max
}
```

сломается на пустом `slice`.

Если пустой вход реален, сигнатура и логика должны это отражать.

---

### Ошибка 5. Путаться в `scope`

Частая проблема:

* переменная объявлена внутри `if`, `for` или функции
* потом код пытается использовать её снаружи

Ещё одна типичная ошибка:

* случайно создать новую переменную через `:=`, а не обновить существующую

---

### Ошибка 6. Путать variadic-функцию и параметр типа `[]T`

Вот это не одно и то же:

```go
func sumSlice(nums []int) int
func sumAll(nums ...int) int
```

Первая функция принимает один `slice`.

Вторая принимает произвольное число аргументов, хотя внутри они и собираются в `slice`.

---

### Ошибка 7. Дробить код на функции без пользы

Маленькие функции — это хорошо, но не нужно превращать код в набор бессмысленных однотипных обёрток.

Плохой сигнал:

* функция ничего не проясняет
* функция только переименовывает одну строчку без реальной пользы
* функция прячет простое действие, но не уменьшает сложность

Хорошее правило:

* выносить шаги, у которых есть понятное имя и отдельная ответственность

---

## 🧠 Полезные паттерны

### 1. Guard clauses и ранний `return`

Пример:

```go
func firstPositive(nums []int) (int, bool) {
	for _, num := range nums {
		if num > 0 {
			return num, true
		}
	}

	return 0, false
}
```

Этот стиль часто проще, чем глубокая вложенность.

---

### 2. Поиск через `(value, ok)` или `(index, ok)`

Примеры:

```go
func findIndex(nums []int, target int) (int, bool)
func findFirstLongerThan(words []string, minLen int) (string, bool)
```

Так сигнатура сразу показывает, что результата может не быть.

---

### 3. Фильтрация как функция, возвращающая новый `slice`

Пример:

```go
func filterEven(nums []int) []int
```

Это хороший базовый стиль на старте:

* входные данные остаются понятными
* результат явно отделён от исходного набора

---

### 4. “Оркестратор + маленькие helper functions”

Например:

* `splitWords`
* `normalizeWords`
* `filterNonEmpty`
* `countWords`

А сверху одна функция или `main`, которая связывает эти шаги.

Так решение читается как последовательность действий, а не как один большой монолит.

---

### 5. Чистые функции как подготовка к тестам

Если функция:

* принимает данные
* возвращает результат
* не печатает сама в консоль
* не зависит от внешнего состояния

её обычно проще тестировать.

Это особенно полезно перед будущим уроком про `go test`.

---

## 🧠 Сравнение с практикой backend

В backend-функции постоянно встречаются, даже если сначала это не так заметно.

Типовые примеры:

* выделить функцию поиска записи по ID
* вынести нормализацию входных строк
* оформить фильтрацию данных в отдельную функцию
* вернуть `(value, ok)` на этапе простого поиска
* позже вернуть `(value, error)` для I/O и бизнес-ошибок

Ещё один важный практический смысл:

* маленькие функции потом легче переносить в отдельные пакеты
* их удобнее покрывать тестами
* review такого кода проходит спокойнее, чем review одной длинной “простыни”

## 🧪 Задачи

### Задача 1 — первая функция `sumTwo`

Напиши функцию:

```go
func sumTwo(a, b int) int
```

Она должна возвращать сумму двух чисел.

Проверь на примерах:

* `sumTwo(10, 20)`
* `sumTwo(-5, 3)`

---

### Задача 2 — сумма элементов `slice`

Напиши функцию:

```go
func sumSlice(nums []int) int
```

Она должна:

* принимать `slice` чисел
* считать сумму всех элементов
* возвращать результат

Проверь на примере:

```go
[]int{4, 7, 1, 8}
```

---

### Задача 3 — поиск индекса

Напиши функцию:

```go
func findIndex(nums []int, target int) (int, bool)
```

Она должна:

* вернуть индекс элемента и `true`, если элемент найден
* вернуть `-1` и `false`, если элемента нет

Подумай:

* почему здесь полезно вернуть `bool`, а не только индекс

### Задача 4 — `minMax` с защитой от пустого `slice`

Напиши функцию:

func minMax(nums []int) (int, int, bool)

Она должна:

* вернуть минимум и максимум
* если `slice` пустой, вернуть `0, 0, false`
* если `slice` не пустой, вернуть `min, max, true`

Проверь минимум на двух случаях:

* непустой `slice`
* пустой `slice`

### Задача 5 — фильтрация значений

Напиши функцию:

func filterGreaterThan(nums []int, limit int) []int

Что важно:

* не пытайся решать задачу прямо в `main`
* результат лучше собирать в новый `slice`

## ⚠️ Важные моменты

* функция должна иметь понятное имя и честную сигнатуру
* несколько результатов в Go — обычный рабочий инструмент
* `(value, ok)` полезен там, где результат может отсутствовать
* `return` завершает функцию сразу
* variadic-параметр внутри функции — это `slice`
* готовый `slice` передаётся в variadic-функцию через `...`
* если решение распадается на несколько логических шагов, лучше дать этим шагам отдельные функции
* маленькие функции — это хороший мостик к пакетам, ошибкам и тестам
