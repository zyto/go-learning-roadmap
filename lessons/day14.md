# 📘 Go | Day 14 — `struct`: поля, литералы, zero value, вложенные структуры, DTO-модели и списки структур

---

## 📖 Теория

### 1. Зачем после указателей переходить к `struct`

На прошлых уроках ты уже разобрал:

* переменные и zero value
* `array`, `slice`, `map`
* `string`, `byte`, `rune`
* функции и несколько return values
* пакеты
* ошибки и `defer`
* указатели и value semantics

Теперь пора собрать отдельные значения в одну понятную модель.

В реальном backend-коде редко работают только с отдельными числами и строками.

Чаще встречаются такие данные:

* пользователь: `ID`, имя, email, активен ли аккаунт
* заказ: номер, сумма, статус
* товар: артикул, название, цена
* ответ API: данные, статус, сообщение

Для таких случаев в Go используется `struct`.

`struct` — это составной тип, который объединяет несколько полей под одним именем.

Главная идея урока:

* научиться объявлять структуры
* создавать значения структур через литералы
* понимать zero value для структуры
* читать и менять поля через точку
* не забывать, что `struct` — это значение и оно копируется
* работать со списком структур через `slice`
* делать поиск, фильтрацию и группировку по полям
* подготовиться к следующему уроку про методы

Короткий словарь:

* `struct` — тип, который состоит из набора полей
* поле — именованная часть структуры, например `Name string`
* литерал структуры — запись вида `User{ID: 1, Name: "Ann"}`
* zero value структуры — структура, у которой каждое поле получило zero value своего типа
* DTO — простая модель данных для передачи между слоями или через границу программы
* вложенная структура — структура, которая лежит в поле другой структуры
* список структур — обычно `[]User`, то есть `slice` значений структур
* поле структуры — не метод; методы будут в следующем уроке

### 2. Первое объявление `struct`

Структура объявляется через `type` и `struct`.

```go
package main

import "fmt"

type User struct {
	ID   int
	Name string
}

func main() {
	var user User

	fmt.Println(user)
}
```

Результат:

```text
{0 }
```

Почему:

* `type User struct { ... }` объявляет новый тип `User`
* внутри структуры есть два поля: `ID int` и `Name string`
* `var user User` создаёт значение типа `User`
* поля не заданы явно, поэтому они получили zero value
* zero value для `int` — `0`
* zero value для `string` — пустая строка `""`

Что здесь важно:

* `User` — это новый тип
* `ID` и `Name` — поля структуры
* структура позволяет хранить связанные данные вместе

---

### 3. Доступ к полям через точку

Поля структуры читаются через точку.

```go
package main

import "fmt"

type User struct {
	ID   int
	Name string
}

func main() {
	user := User{ID: 1, Name: "Anna"}

	fmt.Println(user.ID)
	fmt.Println(user.Name)
}
```

Результат:

```text
1
Anna
```

Почему:

* `user.ID` берёт значение поля `ID`
* `user.Name` берёт значение поля `Name`
* точка означает “обратись к полю внутри этого значения”

Эта запись похожа на доступ к ключу в `map`, но это не одно и то же.

В `map` ключи динамические:

```go
scores["Anna"]
```

В `struct` поля заранее известны компилятору:

```go
user.Name
```

Если написать несуществующее поле, код не скомпилируется.

### 4. Изменение полей

Если структура лежит в переменной, её поля можно менять.

```go
package main

import "fmt"

type User struct {
	ID     int
	Name   string
	Active bool
}

func main() {
	user := User{ID: 1, Name: "Anna"}

	user.Active = true
	user.Name = "Ann"

	fmt.Println(user)
}
```

Результат:

```text
{1 Ann true}
```

Почему:

* сначала `Active` не задан явно, значит он равен `false`
* `user.Active = true` меняет поле `Active`
* `user.Name = "Ann"` меняет поле `Name`

Что здесь важно:

* поле меняется через обычное присваивание
* тип присваиваемого значения должен совпадать с типом поля
* в поле `Name string` нельзя записать число

### 5. Именованный литерал структуры

Самый читаемый способ создать структуру — указать имена полей.

```go
package main

import "fmt"

type Product struct {
	ID    int
	Title string
	Price int
}

func main() {
	product := Product{
		ID:    10,
		Title: "Keyboard",
		Price: 5000,
	}

	fmt.Println(product)
}
```

Результат:

```text
{10 Keyboard 5000}
```

Почему:

* `Product{...}` создаёт значение типа `Product`
* `ID: 10` задаёт поле `ID`
* `Title: "Keyboard"` задаёт поле `Title`
* `Price: 5000` задаёт поле `Price`

Этот стиль удобен, потому что:

* видно, какое значение в какое поле попадает
* порядок полей становится менее опасным
* можно не задавать некоторые поля, если их zero value подходит

---

### 6. Неполный литерал и zero value полей

В именованном литерале можно указать не все поля.

```go
package main

import "fmt"

type User struct {
	ID     int
	Name   string
	Email  string
	Active bool
}

func main() {
	user := User{
		ID:   1,
		Name: "Anna",
	}

	fmt.Println(user.ID)
	fmt.Println(user.Name)
	fmt.Println(user.Email == "")
	fmt.Println(user.Active)
}
```

Результат:

```text
1
Anna
true
false
```

Почему:

* `ID` и `Name` заданы явно
* `Email` не задан, поэтому получил `""`
* `Active` не задан, поэтому получил `false`

Это важная привычка Go:

* не каждое поле нужно обязательно заполнять руками
* иногда zero value — нормальное начальное состояние
* но если zero value опасен или непонятен, лучше создать значение явно

---

### 7. Позиционный литерал структуры

Структуру можно создать без имён полей.

```go
package main

import "fmt"

type Point struct {
	X int
	Y int
}

func main() {
	point := Point{10, 20}

	fmt.Println(point)
}
```

Результат:

```text
{10 20}
```

Почему:

* первое значение попадает в первое поле `X`
* второе значение попадает во второе поле `Y`

Но для учебного и backend-кода обычно лучше именованный литерал:

```go
point := Point{X: 10, Y: 20}
```

Почему:

* меньше риск перепутать поля
* код легче читать
* при добавлении нового поля смысл старого кода понятнее

Позиционный литерал допустим для очень маленьких и очевидных структур, но пока лучше привыкать к именованному стилю.

### 8. `struct` — это значение, и оно копируется

После прошлого урока это должно звучать знакомо.

Структура в Go — обычное значение.

При присваивании она копируется.

```go
package main

import "fmt"

type User struct {
	Name string
	Age  int
}

func main() {
	first := User{Name: "Anna", Age: 30}
	second := first

	second.Name = "Bob"

	fmt.Println(first)
	fmt.Println(second)
}
```

Результат:

```text
{Anna 30}
{Bob 30}
```

Почему:

* `second := first` создаёт копию структуры
* `second.Name = "Bob"` меняет поле только у копии
* `first.Name` остаётся `"Anna"`

Что здесь важно:

* структура не становится “ссылкой” только потому, что в ней несколько полей
* присваивание структуры похоже на присваивание `int`, только копируется больше данных

---

### 9. Параметр функции тоже получает копию структуры

Передача структуры в функцию тоже копирует значение.

```go
package main

import "fmt"

type User struct {
	Name string
	Age  int
}

func rename(user User) {
	user.Name = "Bob"
}

func main() {
	user := User{Name: "Anna", Age: 30}

	rename(user)

	fmt.Println(user)
}
```

Результат:

```text
{Anna 30}
```

Почему:

* `rename(user)` передаёт в функцию копию структуры
* параметр `user` внутри функции — отдельная переменная
* изменение поля внутри функции не меняет исходную структуру в `main`

Если функция должна вернуть изменённую структуру, можно вернуть новое значение.

```go
package main

import "fmt"

type User struct {
	Name string
	Age  int
}

func renamed(user User, name string) User {
	user.Name = name
	return user
}

func main() {
	user := User{Name: "Anna", Age: 30}

	user = renamed(user, "Bob")

	fmt.Println(user)
}
```

Результат:

```text
{Bob 30}
```

Почему:

* функция получает копию
* меняет копию
* возвращает изменённую копию
* `main` явно сохраняет результат обратно в `user`

Этот подход часто хорош для простых преобразований.

---

### 10. Указатель на структуру

Если функция должна изменить исходную структуру, можно передать указатель.

```go
package main

import "fmt"

type User struct {
	Name string
	Age  int
}

func rename(user *User, name string) {
	user.Name = name
}

func main() {
	user := User{Name: "Anna", Age: 30}

	rename(&user, "Bob")

	fmt.Println(user)
}
```

Результат:

```text
{Bob 30}
```

Почему:

* `&user` берёт адрес структуры
* параметр `user *User` получает указатель на структуру
* `user.Name = name` меняет поле исходной структуры

Полная форма была бы такой:

```go
(*user).Name = name
```

Но Go позволяет писать короче:

```go
user.Name = name
```

Это синтаксическое удобство.

Важно:

* `user` всё ещё указатель
* просто при доступе к полю Go сам делает разыменование

### 11. `nil`-указатель на структуру

Указатель на структуру может быть `nil`.

Перед доступом к полям такого указателя нужна проверка.

```go
package main

import "fmt"

type User struct {
	Name string
}

func printName(user *User) {
	if user == nil {
		fmt.Println("no user")
		return
	}

	fmt.Println(user.Name)
}

func main() {
	printName(nil)

	user := User{Name: "Anna"}
	printName(&user)
}
```

Результат:

```text
no user
Anna
```

Почему:

* первый вызов получает `nil`
* функция делает guard clause и выходит раньше
* второй вызов получает адрес реальной структуры
* после проверки можно безопасно читать `user.Name`

Что здесь важно:

* `user.Name` для `nil`-указателя приведёт к `panic`
* проверка `nil` должна идти до доступа к полям
* это тот же принцип, что в прошлом уроке с `*int`

### 12. Структура внутри структуры

Поле структуры может иметь тип другой структуры.

```go
package main

import "fmt"

type Address struct {
	City   string
	Street string
}

type User struct {
	Name    string
	Address Address
}

func main() {
	user := User{
		Name: "Anna",
		Address: Address{
			City:   "Moscow",
			Street: "Tverskaya",
		},
	}

	fmt.Println(user.Name)
	fmt.Println(user.Address.City)
	fmt.Println(user.Address.Street)
}
```

Результат:

```text
Anna
Moscow
Tverskaya
```

Почему:

* `User` содержит поле `Address`
* поле `Address` само является структурой
* `user.Address.City` сначала берёт поле `Address`, потом поле `City`

Это называется вложенная структура.

На текущем уровне лучше использовать явное именованное поле:

```go
Address Address
```

В Go есть ещё встраивание структур без имени поля, но это отдельная тема рядом с композициями и методами.

Сейчас достаточно обычного вложения через именованное поле.

### 13. `slice` структур

Часто нужно работать не с одной структурой, а со списком.

```go
package main

import "fmt"

type User struct {
	ID     int
	Name   string
	Active bool
}

func main() {
	users := []User{
		{ID: 1, Name: "Anna", Active: true},
		{ID: 2, Name: "Bob", Active: false},
		{ID: 3, Name: "Kate", Active: true},
	}

	for _, user := range users {
		fmt.Println(user.ID, user.Name, user.Active)
	}
}
```

Результат:

```text
1 Anna true
2 Bob false
3 Kate true
```

Почему:

* `users` имеет тип `[]User`
* каждый элемент `slice` — значение структуры `User`
* в литерале `[]User{...}` тип элементов уже понятен
* поэтому внутри можно писать коротко `{ID: 1, ...}`, а не `User{ID: 1, ...}`

Что здесь важно:

* `[]User` — это обычный `slice`
* к нему применимы `len`, `append`, `range`
* каждый элемент имеет поля

---

### 14. Осторожно: переменная в `range` — копия элемента

Это важный edge case.

Когда ты пишешь:

```go
for _, user := range users
```

переменная `user` получает копию элемента.

Если менять `user`, исходный элемент в `slice` не изменится.

```go
package main

import "fmt"

type User struct {
	Name   string
	Active bool
}

func main() {
	users := []User{
		{Name: "Anna"},
		{Name: "Bob"},
	}

	for _, user := range users {
		user.Active = true
	}

	fmt.Println(users)
}
```

Результат:

```text
[{Anna false} {Bob false}]
```

Почему:

* `user` внутри `range` — копия очередного элемента
* `user.Active = true` меняет только копию
* исходные элементы `users[0]` и `users[1]` не изменились

Если нужно изменить элементы `slice`, используй индекс.

```go
package main

import "fmt"

type User struct {
	Name   string
	Active bool
}

func main() {
	users := []User{
		{Name: "Anna"},
		{Name: "Bob"},
	}

	for i := range users {
		users[i].Active = true
	}

	fmt.Println(users)
}
```

Результат:

```text
[{Anna true} {Bob true}]
```

Почему:

* `i` — индекс исходного элемента
* `users[i].Active = true` меняет поле прямо в элементе `slice`

Это прямое продолжение прошлого урока:

* `slice` копируется как значение
* но элементы доступны через общий underlying array
* чтобы изменить конкретный элемент структуры, обращайся к нему через индекс

### 15. Поиск структуры в `slice`

Обычная backend-задача — найти запись по полю.

Например, найти пользователя по `ID`.

```go
package main

import "fmt"

type User struct {
	ID   int
	Name string
}

func findUserByID(users []User, id int) (User, bool) {
	for _, user := range users {
		if user.ID == id {
			return user, true
		}
	}

	return User{}, false
}

func main() {
	users := []User{
		{ID: 1, Name: "Anna"},
		{ID: 2, Name: "Bob"},
	}

	user, ok := findUserByID(users, 2)
	fmt.Println(ok)
	fmt.Println(user)

	missing, ok := findUserByID(users, 10)
	fmt.Println(ok)
	fmt.Println(missing)
}
```

Результат:

```text
true
{2 Bob}
false
{0 }
```

Почему:

* функция идёт по `slice` структур
* сравнивает поле `user.ID`
* если нашла, возвращает найденную структуру и `true`
* если не нашла, возвращает zero value структуры и `false`

Паттерн `(value, ok)` уже был в уроках про `map` и функции.

Здесь он снова полезен:

* `User{}` сам по себе может быть валидным значением
* `ok` явно говорит, найден пользователь или нет

### 16. Фильтрация списка структур

Фильтрация структур похожа на фильтрацию чисел или строк.

Разница только в условии: теперь мы смотрим на поля.

```go
package main

import "fmt"

type User struct {
	Name   string
	Active bool
}

func activeUsers(users []User) []User {
	result := make([]User, 0)

	for _, user := range users {
		if user.Active {
			result = append(result, user)
		}
	}

	return result
}

func main() {
	users := []User{
		{Name: "Anna", Active: true},
		{Name: "Bob", Active: false},
		{Name: "Kate", Active: true},
	}

	fmt.Println(activeUsers(users))
}
```

Результат:

```text
[{Anna true} {Kate true}]
```

Почему:

* функция создаёт новый `slice`
* проходит по всем пользователям
* добавляет только тех, у кого `Active == true`
* исходный `slice` не меняется

Это знакомый алгоритм “сбор результата в новый `slice`”, только теперь элементы — структуры.

### 17. Группировка структур по полю

Ещё одна частая задача — сгруппировать записи по значению поля.

Например, товары по категории.

```go
package main

import "fmt"

type Product struct {
	Title    string
	Category string
}

func groupByCategory(products []Product) map[string][]Product {
	result := make(map[string][]Product)

	for _, product := range products {
		result[product.Category] = append(result[product.Category], product)
	}

	return result
}

func main() {
	products := []Product{
		{Title: "Keyboard", Category: "devices"},
		{Title: "Mouse", Category: "devices"},
		{Title: "Desk", Category: "furniture"},
	}

	groups := groupByCategory(products)

	fmt.Println(groups["devices"])
	fmt.Println(groups["furniture"])
}
```

Результат:

```text
[{Keyboard devices} {Mouse devices}]
[{Desk furniture}]
```

Почему:

* ключ `map` — категория
* значение `map` — `slice` товаров этой категории
* если ключа ещё не было, `result[product.Category]` даёт nil slice
* `append` умеет добавлять в nil slice
* поэтому отдельная проверка существования ключа здесь не нужна

Это важный мостик между `map`, `slice` и `struct`.

### 18. Подсчёт и агрегация по структурам

Иногда нужно не вернуть сами структуры, а посчитать итог.

Например, сумму заказов по статусу.

```go
package main

import "fmt"

type Order struct {
	Status string
	Total  int
}

func sumByStatus(orders []Order) map[string]int {
	result := make(map[string]int)

	for _, order := range orders {
		result[order.Status] += order.Total
	}

	return result
}

func main() {
	orders := []Order{
		{Status: "paid", Total: 100},
		{Status: "new", Total: 50},
		{Status: "paid", Total: 70},
	}

	totals := sumByStatus(orders)

	fmt.Println(totals["paid"])
	fmt.Println(totals["new"])
}
```

Результат:

```text
170
50
```

Почему:

* `result[order.Status]` для нового ключа сначала даёт zero value `int`, то есть `0`
* `+= order.Total` прибавляет сумму заказа
* в итоге по каждому статусу накапливается сумма

Слово “агрегация” здесь означает:

* пройти по набору данных
* собрать итоговое значение
* например количество, сумму, максимум или группировку

---

### 19. DTO как простая модель данных

`DTO` расшифровывается как Data Transfer Object.

На практическом уровне пока достаточно такого понимания:

* DTO — это простая структура для переноса данных
* обычно в ней нет сложной логики
* она описывает форму данных: какие поля есть и каких они типов

Пример:

```go
package main

import "fmt"

type CreateUserDTO struct {
	Name  string
	Email string
	Age   int
}

func isValidUser(dto CreateUserDTO) bool {
	if dto.Name == "" {
		return false
	}

	if dto.Email == "" {
		return false
	}

	if dto.Age <= 0 {
		return false
	}

	return true
}

func main() {
	dto := CreateUserDTO{
		Name:  "Anna",
		Email: "anna@example.com",
		Age:   30,
	}

	fmt.Println(isValidUser(dto))
}
```

Результат:

```text
true
```

Почему:

* `CreateUserDTO` описывает данные для создания пользователя
* функция `isValidUser` проверяет поля
* если обязательные поля заполнены, возвращается `true`

В будущих backend-уроках DTO будут встречаться рядом с:

* JSON
* HTTP request / response
* валидацией входных данных
* слоями приложения

Но сейчас DTO — это просто структура с понятным назначением.

Не нужно пока добавлять:

* JSON-теги
* методы
* интерфейсы
* сложные конструкторы

Это будет позже.

### 20. Экспортируемые и неэкспортируемые поля

В уроке про пакеты уже было правило:

* имя с большой буквы экспортируется
* имя с маленькой буквы не экспортируется

Для полей структуры правило такое же.

```go
package main

import "fmt"

type User struct {
	ID   int
	Name string
	age  int
}

func main() {
	user := User{
		ID:   1,
		Name: "Anna",
		age:  30,
	}

	fmt.Println(user.ID)
	fmt.Println(user.Name)
	fmt.Println(user.age)
}
```

Результат:

```text
1
Anna
30
```

Почему:

* внутри того же пакета доступны все поля
* `ID` и `Name` начинаются с большой буквы
* `age` начинается с маленькой буквы

Разница станет важнее, когда структура находится в одном пакете, а используется из другого.

Для будущего backend-кода важно:

* DTO и модели, которые должны использоваться снаружи пакета, часто имеют экспортируемые поля
* внутренние детали можно оставлять неэкспортируемыми

Глубоко проектировать пакеты сейчас не нужно.

Достаточно помнить, что правило большой буквы работает и для полей структур.

### 21. Мостик к методам

Следующий урок будет про методы.

Метод — это функция, привязанная к типу.

Пока не нужно писать методы.

Но важно подготовить интуицию:

```go
type User struct {
	Name string
}
```

Сейчас ты можешь писать обычную функцию:

```go
func printUser(user User) {
	fmt.Println(user.Name)
}
```

В следующем уроке ты увидишь, как похожее действие можно записать как метод у `User`.

Также там станет важным выбор:

* value receiver — метод получает копию структуры
* pointer receiver — метод получает указатель на структуру

Этот выбор опирается на текущий урок:

* структура копируется как значение
* указатель на структуру позволяет менять исходную структуру
* `nil`-указатель требует осторожности

---

## 🔎 Что особенно важно понять

* `struct` объединяет несколько полей в один тип.
* Поле структуры имеет имя и тип.
* Значение структуры можно создать через литерал.
* Именованный литерал обычно безопаснее и понятнее позиционного.
* Если поле не задано явно, оно получает zero value своего типа.
* Доступ к полю идёт через точку: `user.Name`.
* `struct` — это значение, и при присваивании оно копируется.
* Параметр функции типа `User` получает копию структуры.
* Чтобы функция изменила исходную структуру, можно передать `*User`.
* Указатель на структуру может быть `nil`.
* Перед доступом к полям через потенциально `nil`-указатель нужна проверка.
* Вложенная структура — это обычное поле, тип которого тоже структура.
* `[]User` — это обычный `slice`, элементы которого являются структурами.
* В `range` переменная элемента — копия; для изменения элемента используй индекс.
* Поиск, фильтрация и группировка по полям строятся на уже знакомых `for`, `if`, `slice` и `map`.
* DTO — это простая структура для передачи данных.
* Методы, JSON и тесты будут позже; в этом уроке они не нужны.

## ⚠️ Типичные ошибки

### Ошибка 1. Думать, что `struct` передаётся в функцию “по ссылке”

Неверная интуиция:

```go
func rename(user User) {
	user.Name = "Bob"
}
```

Такая функция меняет копию.

Если нужно изменить исходное значение:

* верни изменённую структуру
* или передай указатель `*User`

---

### Ошибка 2. Менять элемент `slice` структур через копию из `range`

Опасная версия:

```go
for _, user := range users {
	user.Active = true
}
```

Так меняется только копия элемента.

Если нужно изменить исходный элемент:

```go
for i := range users {
	users[i].Active = true
}
```

---

### Ошибка 3. Использовать позиционный литерал там, где легко перепутать поля

Плохо читается:

```go
user := User{1, "Anna", true}
```

Лучше:

```go
user := User{ID: 1, Name: "Anna", Active: true}
```

Именованный литерал помогает читать код без знания порядка полей наизусть.

---

### Ошибка 4. Забывать про zero value полей

```go
user := User{Name: "Anna"}
```

Если у структуры есть поле `Active bool`, оно будет `false`.

Это может быть нормально, а может быть ошибкой бизнес-логики.

Всегда проверяй:

* подходит ли zero value для поля
* нужно ли задать поле явно

---

### Ошибка 5. Разыменовывать `nil`-указатель на структуру

Опасно:

```go
func printName(user *User) {
	fmt.Println(user.Name)
}
```

Если `user == nil`, будет `panic`.

Безопаснее:

```go
func printName(user *User) {
	if user == nil {
		fmt.Println("no user")
		return
	}

	fmt.Println(user.Name)
}
```

---

### Ошибка 6. Преждевременно добавлять методы, интерфейсы и JSON-теги

На этом уроке задача — понять сами структуры.

Пока не нужно усложнять код:

* методами
* интерфейсами
* JSON-тегами
* конструкторами “на будущее”

Если структура просто хранит данные, пусть она просто хранит данные.

---

### Ошибка 7. Использовать `map[string]string` вместо понятной структуры

Плохая привычка из динамических языков:

```go
user := map[string]string{
	"id":   "1",
	"name": "Anna",
}
```

Для известных данных лучше структура:

```go
user := User{ID: 1, Name: "Anna"}
```

Почему:

* поля проверяются компилятором
* типы полей ясны
* `ID` остаётся числом, а не строкой
* код легче читать и менять

## 🧠 Полезные паттерны

### 1. Именованный литерал для создания модели

```go
user := User{
	ID:     1,
	Name:   "Anna",
	Active: true,
}
```

Паттерн:

* явно указать поля
* не полагаться на порядок
* оставить zero value только там, где он действительно подходит

---

### 2. Вернуть изменённую структуру, если это простое преобразование

```go
func withName(user User, name string) User {
	user.Name = name
	return user
}
```

Паттерн:

* функция получает значение
* меняет копию
* возвращает результат

Так хорошо делать, когда не нужно менять исходную переменную скрыто.

---

### 3. Передать `*Struct`, если функция должна изменить исходное значение

```go
func activate(user *User) {
	if user == nil {
		return
	}

	user.Active = true
}
```

Паттерн:

* принять указатель
* проверить `nil`, если он возможен
* изменить поле исходной структуры

---

### 4. Для изменения элементов `[]Struct` идти по индексам

```go
for i := range users {
	users[i].Active = true
}
```

Паттерн:

* не менять копию из `range`
* обращаться к исходному элементу через индекс

---

### 5. Для поиска возвращать `(Struct, bool)`

```go
func findUserByID(users []User, id int) (User, bool) {
	for _, user := range users {
		if user.ID == id {
			return user, true
		}
	}

	return User{}, false
}
```

Паттерн:

* вернуть найденное значение
* вторым результатом вернуть `ok`
* если не найдено, вернуть zero value и `false`

---

### 6. Для группировки использовать `map[key][]Struct`

```go
groups := make(map[string][]Product)

for _, product := range products {
	groups[product.Category] = append(groups[product.Category], product)
}
```

Паттерн:

* ключ — поле, по которому группируем
* значение — список структур с таким ключом
* `append` работает даже с nil slice

---

## 🧠 Сравнение с практикой backend

В backend-разработке структуры — один из главных инструментов Go.

Они используются для:

* входных данных
* ответов
* доменных моделей
* промежуточных результатов
* конфигурации
* данных из базы
* данных для JSON

Но важно не перепрыгнуть вперёд.

Сейчас структура — это прежде всего способ сделать данные явными:

```go
type User struct {
	ID    int
	Name  string
	Email string
}
```

Такой код лучше, чем набор несвязанных переменных:

```go
id := 1
name := "Anna"
email := "anna@example.com"
```

И лучше, чем динамическая `map`, если поля известны заранее.

DTO в будущих API будет описывать форму входа или выхода:

```go
type CreateUserDTO struct {
	Name  string
	Email string
}
```

Позже рядом появятся:

* `encoding/json`
* HTTP handlers
* request / response
* validation
* методы
* тесты

Но фундамент один:

* понятно объявить структуру
* корректно создать значение
* не перепутать копию и исходное значение
* уметь обработать `[]Struct`

* проверить минимум один валидный и один невалидный пример

Что тренируется:

* DTO как простая структура данных
* проверка полей
* `early return`

### Задача 16 — мини-практика: каталог товаров

Напиши программу с типом:

```go
type Product struct {
	ID       int
	Title    string
	Category string
	Price    int
	Active   bool
}
```

И функциями:

```go
func findProductByID(products []Product, id int) (Product, bool)
func activeProducts(products []Product) []Product
func totalPrice(products []Product) int
func groupProductsByCategory(products []Product) map[string][]Product
func deactivateProduct(products []Product, id int) bool
```

Требования:

Для `findProductByID`:

* найти товар по `ID`
* вернуть `(Product, true)` или `(Product{}, false)`

Для `activeProducts`:

* вернуть новый `slice` только с активными товарами

Для `totalPrice`:

* вернуть сумму цен всех переданных товаров

Для `groupProductsByCategory`:

* сгруппировать товары по категории

Для `deactivateProduct`:

* найти товар по `ID`
* если найден, поменять его поле `Active` на `false` в исходном `slice`
* вернуть `true`, если товар найден
* вернуть `false`, если товар не найден

Пример данных:

```go
products := []Product{
	{ID: 1, Title: "Keyboard", Category: "devices", Price: 5000, Active: true},
	{ID: 2, Title: "Mouse", Category: "devices", Price: 2500, Active: true},
	{ID: 3, Title: "Desk", Category: "furniture", Price: 12000, Active: false},
	{ID: 4, Title: "Lamp", Category: "furniture", Price: 3000, Active: true},
}
```

Что здесь специально тренируется:

* структура как модель данных
* список структур
* поиск по полю
* фильтрация по полю
* агрегация числового поля
* группировка через `map[string][]Product`
* изменение элемента `slice` структур через индекс

Не используй в этой задаче:

* методы
* интерфейсы
* JSON
* тесты

## ⚠️ Важные моменты

* `struct` нужен, чтобы объединить связанные поля в один тип.
* Поля имеют имена и типы.
* Zero value структуры состоит из zero value всех её полей.
* Именованный литерал делает код понятнее.
* Позиционный литерал зависит от порядка полей.
* Доступ к полю идёт через точку.
* `struct` копируется при присваивании.
* `struct` копируется при передаче в функцию.
* Функция может вернуть изменённую копию структуры.
* Функция может принять `*Struct`, если нужно изменить исходное значение.
* Указатель на структуру может быть `nil`.
* Полная форма доступа через указатель — `(*user).Name`, короткая форма — `user.Name`.
* Вложенная структура — обычное поле со структурным типом.
* `slice` структур обрабатывается обычным `for range`.
* Переменная элемента в `range` — копия, поэтому изменение копии не меняет исходный элемент.
* Для изменения элемента `[]Struct` используй индекс.
* Поиск, фильтрация, группировка и агрегация по структурам строятся на уже пройденных `for`, `if`, `slice`, `map`, функциях и `(value, ok)`.
* DTO пока нужно понимать как простую структуру для передачи данных.
* Методы, JSON, интерфейсы и тесты не нужны для выполнения задач этого урока.
