# Профиль ученика

## Сильные стороны

* Большой опыт коммерческой разработки
* Хорошая база SQL
* Понимание бизнес-логики
* Есть backend-опыт на PHP

## Ограничения и риски

* Go изучается с нуля
* Пока слабое понимание памяти и runtime-поведения Go
* Нет практики конкурентности в Go
* Возможен перенос неидиоматичных привычек из 1С / PHP
* Есть риск тратить слишком много энергии на среду, а не на язык и backend-практику

## Вывод для программы

Обучение должно быть ориентировано не на базовое “научиться программировать”, а на:

* быстрое освоение синтаксиса и идиом Go
* переход к backend-практике как можно раньше
* регулярную проверку понимания через review и вопросы
* акцент на объяснение решений и trade-offs
* раннее формирование инженерных привычек: `go fmt`, `go test`, чтение ошибок компилятора, понятная структура кода
* использование среды как инструмента, а не как отдельной сложности на старте

Приоритет обучения:

* `Go + backend + SQL + testing` важнее, чем глубокое погружение в Linux на старте
* Linux осваивается прикладно и по мере роста задач, а не как отдельная цель раньше времени

# Среда разработки

* Основная ОС: Windows
* Дополнительно: WSL2
* IDE: GoLand / Cursor
* Основной путь проекта для IDE и AI-инструментов: `W:\golang\go-learning-roadmap`

## Стратегия среды

Основной принцип:

> На ранних этапах важнее быстро и стабильно учить Go, чем жёстко привязывать весь процесс к WSL.

Практическая стратегия:

* `Phase 1` и большая часть `Phase 2` могут выполняться полностью в Windows
* `WSL` подключается точечно для Linux-like сценариев и backend-инфраструктуры
* если нужен Codex в GoLand, проект открывается по Windows-пути
* среда должна уменьшать трение, а не усложнять базовую практику

## Правила среды

* учебные упражнения и первые mini-проекты по умолчанию писать и запускать в Windows
* `WSL` использовать для задач, где это действительно даёт пользу:
  `Docker`, `PostgreSQL`, shell-инструменты, Linux-процессы и сигналы, env vars, production-like запуск
* Git можно использовать и из Windows, и из WSL, но в рамках одной рабочей сессии желательно держаться одного окружения
* все учебные артефакты сохраняются в репозитории
* приоритет среды для GoLand + Codex: Windows-путь проекта

# Правила обучения

## Основной принцип

> 70% практика / 30% теория

## Рабочий цикл

1. Определяется тема и учебная цель
2. Даётся короткая теория и набор задач
3. Пользователь пишет код сам
4. Код проходит ревью
5. Закрываются ошибки, пробелы и вопросы
6. Тема считается завершённой только после проверки понимания

Дополнительное правило для lesson-файлов:

* в задачах нельзя использовать конструкции, которые не были введены в теории этого же урока или в ранее завершённых уроках
* архитектурные паттерны и прикладные алгоритмы встраивать в урок только там, где это естественно для темы и помогает практической цели; если неестественно — не добавлять их искусственно

## Ограничения

* не решать задачи полностью без попытки пользователя
* не перепрыгивать через фундаментальные темы без явного запроса
* не считать тему завершённой без практической проверки

## Сквозные инженерные привычки

Начиная с ранних этапов закреплять:

* `go run`
* `go build`
* `go fmt`
* `go test`
* чтение ошибок компилятора и рантайма
* объяснение решения своими словами
* выбор простого решения до появления реальной необходимости в усложнении

## Роль Codex

Codex используется как:

* reviewer
* debugger
* mentor
* explainer
* mock interviewer

При review:

* сначала указывать ошибки, риски и пробелы
* потом объяснять причины и направление исправления

# Программа обучения

Программа делится на 5 фаз:

1. `Phase 1 — Core Go Foundations`
2. `Phase 2 — Backend Basics in Go`
3. `Phase 3 — Data Layer and Persistence`
4. `Phase 4 — Concurrency and Production Patterns`
5. `Phase 5 — Architecture, Testing, Interview Readiness`

## Сквозной трек A — Go Toolchain and Workflow

### Цель

С раннего этапа привыкнуть к базовым инструментам Go, без которых невозможно быстро расти в backend-практике.

### Темы

* `go run`
* `go build`
* `go fmt`
* `go test`
* `go test ./...`
* `go mod init`
* `go mod tidy`
* структура модуля и пакетов
* чтение `go doc` и stdlib-first mindset

### Когда применять

* начинать уже в `Phase 1`
* не выделять в отдельную "теоретическую" фазу, а встраивать в практику
* `go test -race` активно подключить в `Phase 4`

## Сквозной трек B — Linux for Go Backend

### Цель

Освоить Linux не как самоцель, а как рабочую среду backend-разработчика на прикладном уровне.

### Темы

* навигация по файловой системе: `pwd`, `ls`, `cd`
* просмотр файлов и поиск по тексту: `cat`, `less`, `grep`
* процессы и сигналы: `ps`, `kill`, `SIGINT`, `SIGTERM`
* переменные окружения
* сетевые проверки: `curl`
* Docker на базовом уровне
* запуск PostgreSQL локально или в контейнере
* базовые логи и диагностика

### Когда применять

* слегка касаться уже в `Phase 2`
* активно использовать в `Phase 3-5`
* не делать обязательным для каждой задачи из `Phase 1`

## Phase 1 — Core Go Foundations

### Цель

Научиться уверенно писать небольшие и средние программы на Go, понимать базовые структуры данных, функции, методы, ошибки, тесты и ключевые идиомы языка.

### Темы

* синтаксис и управление потоком:
  `if / else`, `switch`, `for`, `break`, `continue`, вложенные циклы
* структуры данных:
  `array`, `slice`, `map`, `string`, `byte`, `rune`
* функции и организация кода:
  параметры, return values, variadic functions, `defer`, scope
* типы и поведение:
  pointers, value semantics, `struct`, методы, pointer/value receiver, базовые интерфейсы
* надежность:
  `error`, wrapping, `errors.Is`, `errors.As`
* устройство кода:
  `package`, экспорт, stdlib-first mindset
* базовый toolchain:
  `go run`, `go build`, `go fmt`, `go test`
* базовое тестирование:
  функция, ожидаемый результат, простые unit tests, первые table-driven tests
* прикладная стандартная библиотека:
  `encoding/json`, `os`, `io`, `bufio`, `time`, `context`, `flag`

### Явная последовательность уроков

Уроки `Day 01`-`Day 05` уже пройдены. Ниже — явная последовательность следующего участка `Phase 1`, чтобы не терять практичные темы и не допускать разрыва между теорией и задачами.

* `Day 06` — `slice`: shorthand-срезы `[:j]`, `[i:]`, `[:]`, `make([]T, len, cap)`, `range`, `nil slice` vs пустой `slice`, `copy`; алгоритмы: фильтрация, копирование коллекции, сбор результата в новый `slice`
* `Day 07` — `map`: создание, чтение, запись, zero value, `ok`-идиома, `delete`, частотный словарь, set-паттерн; алгоритмы: частотный словарь, дедупликация, индексирование данных, базовый `group by`
* `Day 08` — `string`, `byte`, `rune`, `UTF-8`: длина строки, обход, индексация, почему строки нельзя бездумно обрабатывать как массив символов; алгоритмы: реверс строки, подсчёт символов, фильтрация символов
* `Day 09` — stdlib для коллекций и строк: `slices`, `maps`, `strings`, `strconv` на базовом уровне; что уже уметь делать руками циклом и что удобно брать из стандартной библиотеки; алгоритмы: поиск элемента, объединение коллекций, разбиение и сборка строки, сортировка простого `slice`, независимое копирование коллекции
* `Day 10` — функции: параметры, несколько return values, variadic functions, разбиение решения на маленькие функции; алгоритмы: вынесение в функции задач на `sum`, `min/max`, `filter`, `find`
* `Day 11` — `package`, `import`, структура маленького проекта, экспортируемые и неэкспортируемые имена
* `Day 12` — `error` и `defer`: ошибки как значения, early return, `fmt.Errorf`, базовое оборачивание ошибок, безопасное закрытие ресурсов через `defer`
* `Day 13` — указатели и value semantics: когда передаётся копия, когда нужен указатель, как безопасно менять данные
* `Day 14` — `struct`: объявление, литералы, zero value, вложение структур, DTO-подобные модели; алгоритмы: работа со списком структур, поиск и группировка по полям; паттерны: `DTO` как прикладной паттерн модели данных
* `Day 15` — методы: value receiver, pointer receiver, базовое поведение методов на структурах; паттерны: `Factory function` в более осмысленном виде через корректную инициализацию структур
* `Day 16` — интерфейсы: маленькие интерфейсы, implicit implementation, why interface lives on consumer side, без premature abstraction; паттерны: `Strategy`, `Adapter`
* `Day 17` — toolchain и тесты: `go mod`, `go fmt`, `go build`, `go test`, первые unit tests, table-driven tests на базовом уровне; алгоритмы: тестирование фильтрации, частотного словаря, `min/max`, поиска
* `Day 18` — файлы, `io`, `bufio`, `encoding/json`: чтение, запись, простая сериализация и десериализация; алгоритмы: transform/filter/group на данных из файла или JSON; паттерны: `DTO`, при необходимости лёгкий `Facade` вокруг чтения и преобразования данных
* `Day 19` — `time`, `context`, env vars, базовый CLI через `flag`: практичные основы для консольных утилит и будущего backend-кода
* `Day 20` — мини-проект `Phase 1`: консольная утилита с пакетами, ошибками, JSON или файлами, базовыми тестами и review; алгоритмы: интеграция фильтрации, lookup, агрегации, сортировки или группировки по теме проекта; паттерны: `Factory function`, `Facade`, при уместности лёгкий `Adapter`

### Паттерны и антипаттерны

* паттерны:
  early return, guard clauses, small functions, zero value friendly design, composition over inheritance
* антипаттерны:
  лишняя вложенность, premature abstraction, интерфейсы "на будущее", `panic` вместо `error`, размытые имена вроде `utils` / `helpers`

### Алгоритмический минимум

* проход по массиву и `slice`
* суммирование и поиск минимума / максимума
* фильтрация значений
* частотный словарь через `map`
* поиск дубликатов
* реверс строки с учетом `rune`
* двумерный проход через вложенные циклы

### Результат фазы

* 40-50 маленьких упражнений
* 1 консольный мини-проект
* 5-10 простых unit tests
* 2 mini mock interview

## Phase 2 — Backend Basics in Go

### Цель

Научиться писать простой, но корректный backend на Go без магии фреймворков.

### Темы

* `net/http`
* HTTP client
* handlers
* routing
* `http.ServeMux`
* query params / path params
* HTTP status codes
* `encoding/json`
* request / response structure
* request validation
* middleware
* `log/slog`
* logging
* config
* env vars
* graceful shutdown
* `context` в request lifecycle
* первые HTTP-тесты через `httptest`

### Паттерны и антипаттерны

* паттерны:
  `Decorator` через middleware, `Adapter` для внешних клиентов и интерфейсов, `Facade` для service layer, `Dependency Injection`, `Functional Options` для конфигурации сервера и клиента
* антипаттерны:
  жирные handlers, бизнес-логика в HTTP-слое, магические JSON-ответы, размазанная валидация, скрытые side effects, отсутствие контекста и таймаутов в I/O-bound сценариях

### Алгоритмический минимум

* фильтрация коллекций
* дедупликация записей
* пагинация в памяти
* сортировка списка структур
* группировка данных для ответа API
* бинарный поиск
* merge intervals для API и data-сценариев, где это уместно

### Результат фазы

* 1 mini CRUD API без БД
* разбор структуры handler -> service
* review backend-кода
* первые handler/unit tests
* практический навык писать и тестировать небольшой REST API без фреймворка

## Phase 3 — Data Layer and Persistence

### Цель

Научиться работать с PostgreSQL из Go и строить базовый persistence layer.

### Темы

* `database/sql`
* PostgreSQL
* CRUD
* фильтрация
* пагинация
* транзакции
* индексы
* migrations
* repository pattern
* connection pool basics
* `context` в запросах к БД
* интеграционные тесты для data layer
* запуск БД локально или через контейнер

### Паттерны и антипаттерны

* паттерны:
  `Repository`, `Unit of Work` — только если реально понадобится и без форса, `Facade` над storage layer там, где он упрощает работу с несколькими источниками данных
* антипаттерны:
  SQL в случайных местах кода, N+1 мышление, неявные транзакции, смешивание бизнес-логики и SQL-деталей, отсутствие ограничений по времени жизни запросов

### Алгоритмический минимум

* индексация данных через `map`
* grouping
* aggregation
* merge/join нескольких наборов данных
* lookup optimization

### Результат фазы

* CRUD API с PostgreSQL
* минимум один сценарий с транзакцией
* 2-3 SQL-запроса с фильтрацией и пагинацией
* опыт поднятия и подключения локальной БД
* базовые интеграционные тесты для слоя работы с данными

## Phase 4 — Concurrency and Production Patterns

### Цель

Понять конкурентность в Go на уровне, достаточном для уверенной backend-практики.

### Темы

* goroutines
* channels
* buffered / unbuffered channels
* `select`
* mutex
* `sync.WaitGroup`
* `sync.RWMutex`
* `sync/atomic` на базовом уровне
* race condition
* goroutine leaks
* worker pool
* `context` cancellation
* `go test -race`
* benchmarks
* `pprof` на базовом уровне

### Паттерны и антипаттерны

* паттерны:
  `Worker Pool`, `Pipeline`, `Fan-out / Fan-in`
* антипаттерны:
  goroutine leak, shared mutable state без синхронизации, каналы "на всё подряд", игнорирование race conditions, отсутствие backpressure в конкурентной обработке

### Алгоритмический минимум

* batching задач
* merge streams
* coordination patterns

### Результат фазы

* 10-15 упражнений
* 1 прикладной модуль с параллельной обработкой
* разбор типовых race conditions
* практика чтения конкурентных багов и их проверки через `-race`
* первый опыт измерения и профилирования узких мест на базовом уровне

## Phase 5 — Architecture, Testing, Interview Readiness

### Цель

Собрать знания в production-like подход и подготовиться к собеседованию на middle Go backend.

### Темы

* слои
* dependency injection
* clean architecture basics
* testability
* unit tests
* table-driven tests
* integration tests
* границы unit / integration tests
* code review simulation
* backend interview questions
* CI/CD basics
* базовая документация сервиса и README
* безопасность: базовая аутентификация и авторизация в типовых API-сценариях
* performance tuning basics

### Паттерны и антипаттерны

* паттерны:
  `Dependency Injection`, `Facade`, `Decorator`, `Strategy`, иногда `Builder` и `Functional Options`
* антипаттерны:
  архитектура ради архитектуры, избыточные интерфейсы на CRUD, mocks everywhere, premature generalization, CI/CD без понимания процесса сборки и тестирования

### Алгоритмический минимум

* отдельный interview block по алгоритмам и структурам данных
* повторение и осознанная практика уже изученных алгоритмов в формате собеседования

### Результат фазы

* 2 проекта в портфолио
* серия mock interview
* финальный список слабых мест и план добивки
* готовность объяснять не только код, но и практические инженерные решения вокруг него

# Формат завершения темы

Тема считается завершённой, только если:

1. Код написан самостоятельно
2. Код прошёл review
3. Решение можно объяснить своими словами
4. Пройдены 3-5 коротких вопросов по теме
5. Нет критической путаницы в базовых edge cases
